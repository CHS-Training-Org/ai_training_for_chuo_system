# create-issue — GitHub Issue を聞き取って実起票する

ユーザーからインタビュー形式で内容を聞き取り、Issue テンプレートの様式に沿った Markdown 本文を組み立て、**`gh` CLI で `CHS-Training-Org/ai_training_for_chuo_system` に実際に Issue を起票する**スキル。本文の下書きで終わらせず、プレビュー確認を経て起票まで行い、作成された Issue の URL を返す。

対応する起票種別は 2 つ:

- **選択課題（エンハンス）** — `.github/ISSUE_TEMPLATE/optional-task.yml` 準拠
- **汎用 Issue** — リポジトリ既定テンプレートには無い一般様式

必須ステップ（STEP-01〜03）は GitHub Issue を起票しない（`Docs/guide/curriculum.md` §必須ステップ課題を参照）。

> **やること**：聞き取り → 本文組み立て → プレビュー確認 → `gh issue create` 実行 → URL 提示。
> **下書きだけが欲しい場合**は create-pr ではなく、このスキルの「§5 起票前の確認」までで止めればよい（起票してよいか確認するゲートがある）。

---

## 1. 前提チェック（最初に必ず実行）

起票は外向き・不可逆操作なので、`gh` が使えることを最初に確かめる。

```bash
gh auth status
```

- `gh` コマンドが見つからない → インストールを案内して中断する（Windows なので `winget install --id GitHub.cli`）。インストール後にシェルの再起動が必要な点も伝える。
- `gh` はあるが未認証 → `gh auth login` を案内して中断する。
- 認証済み → 続行する。

`gh` が使えないまま聞き取りだけ進めても起票できないので、**聞き取りより前にこのチェックを行う**。

---

## 2. 対話フロー

1. **種別を選択**してもらう（§3）。
2. 種別ごとの項目を聞き取る（§4）。テンプレート YAML の `body` 項目をそのまま質問にする。
3. 聞き取った内容で **Markdown 本文を組み立てる**（§4 のマッピング）。
4. 本文・タイトル・付与ラベルの **プレビューを提示し、起票してよいか確認**する（§5）。
5. 確認が取れたら `gh issue create` を実行する（§6）。
6. 作成された **Issue の URL を提示**し、メンターによる追加ラベル付与の運用を案内する（§7）。

---

## 3. 種別の選択

`AskUserQuestion` で 2 択を提示する（`.claude/rules/aidlc-questions.md` の様式に従う）:

- **選択課題（エンハンス）** — Beginner / Intermediate / Advanced のエンハンス課題
- **汎用 Issue** — 上記テンプレートに当てはまらない一般的な Issue

---

## 4. 種別ごとの聞き取り項目と本文マッピング

各項目を聞き取り、**見出し付き Markdown** に再構成する。`gh` はフォーム YAML を使えないため、テンプレートの各フィールドを `### <ラベル>` の見出しブロックとして本文に並べる。

未入力の任意項目は本文に含めない（空見出しを残さない）。受入条件・完了条件の詳細は **ドキュメント側が真実の源**なので、本文に再掲しない（§8 Gotchas）。

### 4.1 選択課題（エンハンス）

| 項目 | 必須 | 形式 |
|------|------|------|
| ビジネス要求シートへのリンク | ○ | テキスト（例 `Docs/spec/enhancements/<short-desc>.md`） |
| ゴール | ○ | 複数行 |
| 依存関係（下記手順で自動転記） | – | 複数行 |
| 完了条件（受入条件の詳細は再掲しない・自己チェック用の要約） | ○ | 複数行 |
| AI活用ポイント | – | 複数行 |
| 推定工数 | ○ | 半日 / 1日 / 2〜3日 / 1週間以上 |
| 難易度 | – | Beginner / Intermediate / Advanced |

- タイトル: `[Enhance] <概要>`
- ラベル: `--label "課題" --label "難易度：<選択値>"`（難易度が未入力の場合は `難易度：*` を省略する）
  - 初級 → `難易度：初級`、中級 → `難易度：中級`、上級 → `難易度：上級`

#### シートの存在確認（起票前に必須）

ビジネス要求シートへのリンクを受け取ったら、他の項目を聞く前に必ずファイルの存在を Read（または Glob）で確認する。

- **存在する** → 通常どおり聞き取りを続ける（依存関係の自動転記は次項の手順に従う）。
- **存在しない**（パスのタイポ・シート未作成等） → **ここで起票を中断**し、`/update-spec` スキルで先にビジネス要求シートを作成するよう案内する。Spec-first の原則上、シートが無いまま Issue だけ先行して起票すると、後で要件を詰めた際に Issue 側が追従されず内容が乖離するため。

#### 依存関係欄への転記（必須手順）

ビジネス要求シートには **依存関係** 節（前提課題／競合する課題／推奨着手順序）があり、複数の学習者が並行着手する際のマージ競合を避けるための情報が書かれている（`Docs/spec/enhancements/index.md` 参照）。受入条件・完了条件と違い、この情報は Issue を開いた学習者がその場で気づけないと意味がない（シートを別途開いて確認する運用ではこの節の目的を果たせない）ため、**依存関係節の文章を要約し直さず `依存関係` 欄にほぼそのまま転記してよい**（§8 の「詳細再掲禁止」の対象外）。

ただし依存関係節の Markdown リンク（`[タイトル](./file.md)` 形式）は GitHub Issue 本文の中では相対パスとして解決されず壊れたリンクになる（Issue のレンダリングはリポジトリのファイルツリーではなく Issue の URL を基準にするため）。そのため、**地の文はそのままコピーしつつ、リンク部分だけ**次の変換をかける。

ビジネス要求シートへのリンクを受け取ったら、`依存関係` を聞く前に必ず次を行う:

1. 該当ファイルを Read し、依存関係節を確認する。
2. 依存関係節（前提課題・競合する課題・推奨着手順序のすべて）に含まれる `[タイトル](./file.md)` 形式のリンクごとに、対応する GitHub Issue が既に起票済みか `gh issue list --search "<ファイル名>" --state all` で確認する。
   - 見つかれば `[タイトル](./file.md)` を `タイトル（#123）` のように Issue 番号の記載へ置き換える。
   - 見つからなければ、`[タイトル](./file.md)` というリンクのまま残さず「タイトル（`Docs/spec/enhancements/file.md`）」のようにファイルパスをインラインコードで併記する平文に変換する（起票されていない課題のためにダミーの Issue 番号を作らない。また、壊れるリンクをそのまま Issue に残さない）。
3. 「前提課題：なし」のように依存が無い旨だけの記述は転記しない（空見出しを残さない原則と同じ理由）。
4. リンク以外の文章（説明文、「各種フィルタ課題 等」のような総称的な記述、Flyway マイグレーション番号衝突の説明等）は要約・言い換えせずそのままコピーする。
5. ユーザーが口頭で伝えた前提条件（STEP-01〜03 の完了状況等）があれば、転記した依存関係と同じ `依存関係` ブロック内にまとめる。新しい見出しは追加しない。
6. シートが Read できない（リンクが誤り等）場合は転記をスキップし、その旨をユーザーに伝える（本来は前項「シートの存在確認」で捕捉済みのはずだが、リンク先が spec ファイルではない等の想定外ケースへのフォールバック）。

出力イメージ（`### 依存関係` ブロック内）:

```
STEP-03 まで完了していること。

ビジネス要求シートより：
- 競合する課題：部署ごとの承認者設定（#42）。両課題とも `ApprovalService.createStep()` を別方向に拡張するため、並行着手は非推奨。一方を完成させてからもう一方に着手する。
- 推奨着手順序：本課題の完成後に既存機能の E2E テスト追加（#57）で承認フローのリグレッションをカバーするとよい。
```

依存関係節・ユーザー入力のいずれも無い場合は、`依存関係` ブロックごと本文から省く（§4 冒頭の未入力任意項目の扱いに従う）。

### 4.2 汎用 Issue

リポジトリ既定テンプレートには無い形式。一般様式で聞き取る:

- 概要 / 背景
- 詳細（やること、または再現手順）
- 期待する結果 / 完了の定義
- 補足（任意）

**ラベルは必須**（最低 1 つ）。`gh label list` で取得した実在ラベルを提示し、`AskUserQuestion`（複数選択可）で選ばせる。このリポは自由記述 Issue を嫌う運用（`config.yml` で Web UI の blank issue を禁止）なので、gh 起票でも無ラベルにせず性質を表すラベルを付ける。

現在の実在ラベル（`gh label list` で確認済み）:

| ラベル名 | 用途 |
|---------|------|
| `課題` | 学習課題（選択課題・エンハンス課題） |
| `ドキュメント改善` | Docs や仕様書の誤り・改善提案 |
| `運用・管理` | メンター・オーナーによる設定変更・整備タスク |
| `バグ報告` | 学習環境・コード・ドキュメントの不具合報告 |
| `質問・相談` | 学習者からの質問・詰まり相談 |
| `難易度：初級` / `難易度：中級` / `難易度：上級` | 難易度（スキルが自動付与 — 後述） |
| `対象：フロントエンド` / `対象：バックエンド` / `対象：フルスタック` | 対象レイヤー（メンター運用） |
| `進行中` | 着手中（メンター運用） |

選んだものを `--label "<name>"` で 1 つずつ渡す。

**難易度の付与**：`AskUserQuestion` で難易度（初級 / 中級 / 上級 / 指定しない）を選ばせ、指定があれば `--label "難易度：<選択値>"` を自動付与する。難易度が Issue の性質に合わない場合（ドキュメント改善・運用整備等）は「指定しない」を選択させる。

| 難易度選択 | 付与するラベル |
|-----------|-------------|
| 初級 | `難易度：初級` |
| 中級 | `難易度：中級` |
| 上級 | `難易度：上級` |
| 指定しない | ラベル付与なし |

**タイトルのプレフィックス**：選択した「性質ラベル」に応じてタイトルの先頭にプレフィックスを付ける。複数の性質ラベルを選んだ場合は最も主たるものを 1 つ選ぶ。

| 選択ラベル | タイトルプレフィックス |
|-----------|----------------------|
| `ドキュメント改善` | `[ドキュメント]` |
| `運用・管理` | `[運用]` |
| `バグ報告` | `[バグ]` |
| `質問・相談` | `[質問]` |
| `課題` | プレフィックス不要（Enhance 種別で設定済み） |

> タイトルはユーザーから受け取った概要テキストの先頭にプレフィックスを自動付与して提示する。確認ゲート（§5）でプレビューし、修正を受け付ける。

## 5. 起票前の確認（必須ゲート）

起票は外向き・不可逆なので、実行前に必ず止める。以下をまとめて提示し、起票してよいか確認する:

- **タイトル**
- **付与する `--label`**（複数あれば全部）
- **本文の全文**（組み立てた Markdown）

ユーザーが承認したら起票に進む。修正要望があれば本文を直して再提示する。

## 6. gh で起票する

本文は **`--body-file` で渡す**。インライン `--body "..."` は PowerShell のクォート・改行で壊れやすい。本文を scratchpad に一時ファイルとして書き出し、それを渡す。

PowerShell が主シェルなので、コマンドは1行（または PowerShell のバックティック継続）で組み立てる。bash のバックスラッシュ継続は使わない。

```
gh issue create --title "[Enhance] リソース一覧の検索・フィルタ追加" --label "課題" --body-file <scratchpad>/issue-body.md
```

- ラベルが複数のときは `--label "a" --label "b"` のように繰り返す。
- **テンプレートの自動ラベル付与は gh では効かない**ため、`課題` / `ドキュメント改善` 等も含めて `--label` を必ず明示する（`Docs/guide/issue-registration.md` §howto の注意書きと整合）。
- 成功すると gh が Issue の URL を標準出力に返す。
- **未登録ラベルでの失敗に備える**：`--label` のラベル実体がリポジトリに無いと gh は `could not add label: 'X' not found` 等で**エラーになり issue は作られない**（§8）。このエラーが出たら、勝手にラベルを外して再起票せず、label-sync 未実行の可能性をユーザーに伝えて対処を促す（`Docs/guide/issue-registration.md` §label-sync）。心配な場合は起票前に `gh label list` で実体を確認してもよい。

---

## 7. 起票後の案内

- 作成された **Issue の URL** を提示する。
- メンターが運用する追加ラベル（`対象：*` / `進行中`）は**このスキルでは付けない**。起票後にメンターが [§ラベルマッピング規則](../../../Docs/guide/issue-registration.md#mapping) に従って手動付与する旨を一言添える。
- `難易度：*` はスキルが起票時に自動付与するため、メンター手動付与は不要。

## 8. Gotchas

- **gh 起票はテンプレートのフォーム・自動ラベルをバイパスする** → `課題` / `ドキュメント改善` 等も `--label` で明示しないと付かない。
- **ラベル実体が無いときの挙動は経路で異なる**：
  - Web テンプレートの自動付与は、ラベル実体が無いと**サイレントに無視**される（label-sync 前提。`Docs/guide/issue-registration.md` §label-sync）。
  - **gh の `--label` はサイレントに無視されず、ハードエラーになる**（`could not add label: 'X' not found` 等）。gh はラベル名を API で解決してから起票するため、未登録ラベルを指定すると **issue 自体が作られず失敗する**。起票前に `gh label list` でラベル実体を確認すること（§4.2 の表は最後に確認した実在ラベル一覧だが、変更されている可能性があるため都度確認が望ましい）。
- **PowerShell のクォート問題** → 本文は必ず `--body-file` で渡す。インライン `--body` は使わない。
- **汎用 Issue と「自由記述禁止」方針の緊張関係**：`config.yml` は `blank_issues_enabled: false` で Web UI の自由記述を禁止しているが、**gh 起票はこれをバイパスできる**。汎用 Issue を作る際はこの点を意識し、ラベル付与（§4.2）で性質を明示して運用方針と齟齬が出ないようにする。
- **完了条件・受入条件はドキュメント側が真実の源**。Issue 本文に詳細を再掲しない（二重管理防止。テンプレートの注記と整合）。**依存関係（前提課題・競合する課題・推奨着手順序）はこの原則の例外**で、並行着手時の衝突回避のため `依存関係` 欄への転記が必須（リンクは Issue 番号表記へ変換する。§4.1「依存関係欄への転記」）。
- **メンター運用ラベルを勝手に付けない**：`対象：*` / `進行中` はメンターが付与する。`難易度：*` はスキルが自動付与するため、メンター付与は不要（§4.1〜4.2 の難易度選択に従う）。
- **タイトルプレフィックスは性質ラベルから自動決定する**（§4.2 の対応表）。プレフィックス未設定のまま起票しない。起票前の確認ゲート（§5）でプレビューに含めること。

## 9. 質問様式

聞き取りは2系統を使い分ける:

- **離散的な選択（選択肢が決まっているもの）は `AskUserQuestion`**：種別選択（§3）、推定工数（半日/1日/2〜3日/1週間以上）、難易度（Beginner/Intermediate/Advanced）、汎用 Issue のラベル選択（§4.2）など。`.claude/rules/aidlc-questions.md` 準拠で1回最大4問・相互排他・意味のある選択肢のみ。選択肢が3つ超に分岐するなら質問を分割する。
- **自由記述の項目は会話で聞き取る**：ゴール・依存関係・完了条件・各種リンク・概要・詳細などは決まった選択肢が無いので、`AskUserQuestion` に無理に押し込まず、通常の会話でテキストとして尋ねる。

曖昧な回答や矛盾は、進める前に追加質問で解消する。
