# create-pr — PR を組み立てて、必要なら作成する

`.github/PULL_REQUEST_TEMPLATE.md` が BookFlow の PR 文面の様式（真実の源）。このスキルはその様式に沿って head/base 間の変更内容から PR タイトル・本文を組み立てる。`gh` が使える環境では、確認のうえ `gh pr create` まで実行して PR の URL を返す。`gh` が使えない環境や、ユーザーが下書きだけを望む場合は、本文を提示するところで止め、手動で GitHub の PR 作成画面に貼り付けてもらう。

**やらないこと**：コミットの分割・`git push` の実行。これは `commit-push` スキルの責務であり、実際に PR を作成するモードでは head ブランチが既に origin に push 済みであることを前提にする。

---

## 1. ブランチと実行方法を確認する（最初にまとめて確認）

PR の元になる差分は head/base の組み合わせで決まり、実行方法（下書きだけか作成まで行うか）によって以降の手順が変わる。両方とも本文を組み立てる前に確認する。

まず `gh auth status` を実行し、`gh` が使えるか（インストール済みかつ認証済みか）を確かめる。使えない場合は「実際に作成する」を選択肢から外し、理由（`gh` が未インストール、または未認証）をユーザーに伝えたうえで下書きのみのモードで進める。

そのうえで `AskUserQuestion` で以下をまとめて確認する（`.claude/rules/aidlc-questions.md` 準拠）：

- **head ブランチ（どこから）**：現在のブランチ（`git branch --show-current` で取得した名前）を使うか、別のブランチを指定するか。「別のブランチを指定する」を選んだ場合は、ブランチ名は決まった選択肢が無いので通常の会話で聞く。
- **base ブランチ（どこへ）**：`main` を使うか（BookFlow の既定）、別のブランチを指定するか。同様に、別のブランチの場合は会話で聞く。
- **実行方法**：「本文の下書きだけ提示する（手動で GitHub の PR 作成画面に貼り付ける）」か「確認のうえ `gh pr create` で実際に作成する」か。`gh` が使えない場合はこの選択肢自体を提示しない。

実行方法が「作成する」の場合のみ、head ブランチが push 済みかを確認する：

```bash
git status -sb
git rev-parse --abbrev-ref --symbolic-full-name <head>@{u}
```

追跡ブランチが無い、またはローカルのコミットがリモートより進んでいる（`ahead` が 1 以上）場合は、**push が済んでいない**ということなので、ここで作成に進まず、まず push を終えるよう伝える（`commit-push` スキルの利用を提案してよい）。push は create-pr の責務ではないため、代わりに実行しない。

## 2. 差分を把握する

1. `.github/PULL_REQUEST_TEMPLATE.md` を読み、様式（セクション構成）を確認する。
2. §1 で確定した head/base の差分を把握する：
   - `git log <base>..<head> --oneline`（コミット履歴）
   - `git diff <base>...<head> --stat`（変更ファイル一覧）
   - 必要に応じて個別ファイルの diff も確認する

## 3. タイトル・本文を組み立てる

**タイトル**：`実装概要`（後述）の内容を一文に要約したものを基本とする。対応する Issue 番号が特定できれば末尾に `(#<issue番号>)` を付ける。BookFlow の PR タイトルに固定のプレフィックス規約は無いので、内容が一目でわかる自然な日本語の一文にする。

**本文**：以下の各セクションを埋めた Markdown 全文を組み立てる。

- **対応 Issue / ビジネス要求シート**：head ブランチ名 `feature/<GitHubユーザー名>/<issue番号>-<short-desc>` から Issue 番号を抽出して `Closes #<番号>` を記載する。ビジネス要求シート（`Docs/spec/enhancements/<課題>.md`）への言及がコミットメッセージや diff にあれば、そのパスをリンクとして補う。判別できない場合は空欄のまま残し、ユーザーに確認を促す。
- **実装概要**：`git diff <base>...<head> --stat` と主要なコミットメッセージから、何を変更したか（FE/BE どちらに触れたか含む）を簡潔にまとめる。
- **AI ツールを使った箇所（必須）**：「Workflow Planning で提示された実装計画」「補完・解説・テスト／ドキュメント生成等で AI を使った箇所」は、このセッションでの作業内容から要約して埋めてよい。
- **Spec-first チェック**：`git log <base>..<head> --oneline` の**先頭コミット**が `docs(spec): ...` 形式かを確認する。
  - 先頭が `docs(spec):` → 「仕様変更がある場合、仕様更新を PR の先頭コミットにした」にチェック。
  - `Docs/spec/` への変更が diff に一切ない → 「このPRに仕様変更はない」にチェック。
  - `Docs/spec/` への変更はあるが先頭コミットが `docs(spec):` でない → どちらにもチェックを入れず、ユーザーに注意喚起する（[仕様更新ルール（Spec-first）](../../../Docs/spec/index.md#spec-first) 参照）。
- **動作確認結果**：**AI は記入しない**。テンプレートのプレースホルダーのままにし、「実際に確認した結果を書いてください」とユーザーに伝える。
- **セルフレビュー**：チェックは入れず、[coding-conventions.md §コミット・PR 規約](../../../Docs/guide/coding-conventions.md#commit-pr) の「PR 提出前のセルフレビュー」を見て自分で確認するようユーザーに案内する。
- **任意メモ（メンターへ・あれば）**：実装中に判断に迷った点・設計上のトレードオフがあれば記載する。なければ空欄でよい（メンターレビューは必須ではない）。

## 4. 下書きを提示する

組み立てたタイトルと本文全文を提示する。

- §1 で「下書きだけ」を選んでいた場合：ここで提示して終了する。手動で GitHub の PR 作成画面に貼り付けるよう案内する。
- §1 で「作成する」を選んでいた場合：提示したうえで、この内容で作成してよいか最終確認する。PR 作成は一度実行すると通知やレビューコメントが付き得る操作なので、ここで必ず立ち止まる。修正要望があれば本文を直して再提示する。承認が得られたら §5 に進む。

## 5. `gh pr create` で作成する（§1 で「作成する」を選んだ場合のみ）

本文は **`--body-file` で渡す**。インライン `--body "..."` は PowerShell のクォート・改行で壊れやすい。本文を scratchpad に一時ファイルとして書き出し、それを渡す。

```
gh pr create --title "予約キャンセル操作を追加 (#42)" --body-file <scratchpad>/pr-body.md --base <base> --head <head>
```

- `--base` は §1 で確定したブランチを明示する（省略すると既定ブランチに向いてしまう）。head が現在のブランチと同じ場合は `--head` を省略してよいが、別ブランチを指定された場合は明示する。
- **`--draft` は使わない**。BookFlow はセルフレビュー後に学習者自身がマージする運用（[dev-workflow.md §8](../../../Docs/guide/dev-workflow.md)）なので、GitHub の Draft PR 状態のまま作ると自分でマージできず後から `gh pr ready` が要る。最初から通常の PR として作成する。
- 成功すると `gh` が PR の URL を標準出力に返す。**作成された PR の URL をそのままユーザーに提示する。**

## 6. 案内

- 「動作確認結果」「セルフレビュー」はテンプレートのプレースホルダーのままなので、ユーザー自身が実際に確認した内容を書き込む必要があることを伝える。
- 実際に作成した場合は、マージは学習者自身が行う運用（メンターレビュー必須ではない）ことを案内する。

## Gotchas

- **`gh` が使えない場合は自動的に下書きのみにフォールバックする**：§1 で理由（未インストール／未認証）を伝え、「作成する」の選択肢自体を提示しない。無理に `gh pr create` を試みてエラーにしない。
- **push されていない head ブランチでは作成しない**：§1 で push 済みかを確認し、未 push なら作成に進まず先に push を案内する。create-pr が代わりに `git push` を実行することはない。
- **`--draft` は使わない**：セルフマージ運用と噛み合わないため（§5）。
- **本文は必ず `--body-file` で渡す**：インライン `--body` は PowerShell のクォート・改行で壊れやすい。
- **head/base はブランチが決まっている前提で早合点しない**：現在のブランチ・`main` が既定候補ではあるが、§1 で毎回確認する。
- **完了条件・受入条件はビジネス要求シート／Issue 側が真実の源**：本文に詳細を再掲しない（テンプレートの注記と整合）。
- **動作確認結果・セルフレビューは AI が埋めない**：実際に確認した学習者本人の申告が必要な項目。

## 質問様式

head/base ブランチの選択と実行方法（下書きのみ／作成する）は離散的な選択なので `AskUserQuestion` でまとめて聞く（`.claude/rules/aidlc-questions.md` 準拠：意味のある選択肢のみ・最大4問）。ブランチ名の自由記述部分は通常の会話で聞く。本文の内容修正のような自由記述の要望も同様に会話で受け取る。
