# commit-push — 差分を分割してコミット・push する

作業ツリーの差分を眺めて「これは複数の関心事が混ざっている」と気づいたときに、意味のある単位に分けてコミットし、必要なら push まで行うスキル。**実行前に分割案・ブランチ・push有無をユーザーに確認するゲートを必ず通す**（`git commit`・`git push` は不可逆ではないが、push は共有状態に影響するため）。

**やらないこと**：PR本文の下書き（→ `create-pr`）、Issue起票（→ `create-issue`）、force push・`--amend`・`--no-verify` などの安全プロトコル逸脱（トップレベルの Git Safety Protocol を常に優先する）。

---

## 1. 現在の状態を把握する

```bash
git status
git diff
git diff --staged
git branch --show-current
git log --oneline -5
```

- untracked ファイルも `git status` で拾い、分割対象に含めるか判断する。
- `.env` や credentials など、意図せず含まれていそうなファイルがあれば、分割案を作る前にユーザーに確認する（システムプロンプトの Git Safety Protocol と同じ理由）。

## 2. 分割案を作る

diff を意味のある単位（レイヤー、機能、関心事）ごとにグルーピングし、それぞれに Conventional Commits 形式（`feat:` `fix:` `docs:` `chore:` `refactor:` `test:` `build:` 等）のコミットメッセージ案を付ける。BookFlow のコミット規約は CLAUDE.md 参照。

判断の目安：
- **レイヤーが違えば基本は別コミット**（例: `docs/` の変更と `.devcontainer/` の設定変更は別の関心事）。
- 同じレイヤーでも意味的に無関係な変更（例: typo修正とロジック変更）は分ける。
- 逆に、1つの変更を実現するために複数ファイルにまたがる密接な変更（例: フロントエンドの型定義とそれを使うコンポーネント）は無理に分割しない — 縦切り実装の単位を壊さない。
- 分割しすぎても意味がない。「後で `git log` を読む人が各コミットの意図を一目で理解できるか」を基準にする。

各グループについて、含めるファイルパスの一覧とコミットメッセージ案（タイトル + 必要なら本文）をまとめておく。次のステップでこの案をそのままユーザーに提示する。

## 3. ブランチ・分割案・push有無を一括確認する

`.claude/rules/aidlc-questions.md` の様式に従い、`AskUserQuestion` で以下をまとめて確認する（1回・最大4問）。

- **ブランチ**：現在のブランチのままにするか、新規ブランチを作成するかを毎回問う（現在のブランチが `main` かどうかに関わらず）。新規作成を選んだ場合、ブランチ名 `feature/<GitHubユーザー名>/<issue番号>-<short-desc>`（CLAUDE.md 規約）に必要な issue 番号・短い説明は `AskUserQuestion` に無理に押し込まず、通常の会話で聞く（GitHubユーザー名は `gh api user --jq .login` で取得できる）。
- **分割案**：§2 で作った分割案（グループごとのファイル一覧・コミットメッセージ案）を提示し、「この案でよい」「1つのコミットにまとめる」「一部修正したい」から選ばせる。修正したい場合は自由記述で受け取り、案を作り直してから再度この確認に戻る。
- **push有無**：コミット後に push するかどうかを確認する。

3つとも同じタイミングで確認し、後から一つずつ聞き直さない（分割案の全体像が見えないと push 判断もできないため）。

## 4. 実行する

1. 新規ブランチが選ばれた場合：`git checkout -b <branch-name>`
2. 分割案の順に、グループごとに:
   - `git add <group-1のファイルを明示的に列挙>`（`git add -A` / `git add .` は使わない — 他グループのファイルを巻き込むため）
   - コミット実行。メッセージは HEREDOC で渡す（トップレベルの「Committing changes with git」節の作法に従う。Co-Authored-By の付与もそこに準拠する）
3. 全コミット後に `git log --oneline` と `git status` で結果を確認し、ユーザーに提示する。

pre-commit フックが失敗した場合は原因を直してから**新しいコミットとしてやり直す**（`--amend` は使わない。フック失敗時は直前のコミットが成立していないため、`--amend` だと1つ前の別のコミットを書き換えてしまう）。

## 5. push する（選択された場合のみ）

まずプレーンな `git push` を試す（新規ブランチなら `git push -u origin <branch-name>`）。

失敗した場合、エラー内容が認証由来（`Authentication failed` / `could not read Username` / `Permission denied` 等）であれば、`gh auth status` で `gh` の認証状態を確認し、`gh auth setup-git` で git の credential helper として `gh` の認証情報を設定してから、もう一度だけ `git push` を再試行する。

- 認証由来でない失敗（non-fast-forward、ネットワークエラー等）では `gh` フォールバックを試さず、エラー内容をそのままユーザーに提示する。特に non-fast-forward はリモートに自分の知らないコミットがある可能性があり、`--force` で解決しようとしない。
- `gh` フォールバック後も失敗する場合は、そこで止めてユーザーに状況を報告する。force push や `--no-verify` で無理に通そうとしない。

## Gotchas

- **`git add -A` / `git add .` を使わない**：分割の意味が壊れる。各グループのファイルを明示的に指定する。
- **ブランチ確認は毎回行う**：`main` ブランチ上での作業に気づかず直接コミットしてしまうことを防ぐため、現在のブランチが何であれ確認する。
- **push は認証エラー以外でフォールバックしない**：non-fast-forward を `gh` 経由で解決しようとしても意味がない（認証の問題ではないため）。リモートに新しいコミットがある場合は、まずユーザーに `git fetch` してから状況を確認してもらう。
- **分割案の確認より前に手を動かさない**：diff の把握・分割案作成はいくらでも試行錯誤してよいが、`git add` / `git commit` を実行するのは §3 の確認が取れてから。
- **PR作成はこのスキルの範囲外**：push まで完了したら、必要に応じて `create-pr` スキルの利用を案内する。

## 質問様式

離散的な選択（ブランチを新規作成するか、分割案でよいか、push するか）は `AskUserQuestion` でまとめて聞く。ブランチ名に必要な issue 番号・短い説明のような自由記述項目は、通常の会話で聞く。曖昧な回答や矛盾する回答は、進める前に追加質問で解消する。
