# spec-conventions — BookFlow 仕様書の表記規約・実例集

SKILL.md の手順 2 で「対象ファイルを既存規約に合わせて書く」ときに参照する。各ファイルの実物のフォーマットをテンプレート化したもの。

---

## 明示アンカー `{ #id }` の付与基準

Zensical（`markdown_extensions.toc`・`permalink = true`）は ASCII 見出しには自動アンカーを生成するが、**日本語見出しには安定したアンカーを生成しない**。そのため**他所からリンクされる日本語見出しには明示アンカー `{ #id }` を必ず付ける**。

書き方（見出し行末にスペース区切りで付ける）：

```markdown
### UC-05：承認必要リソースの予約が承認者に回覧される { #uc-05 }
### 日時フォーマット { #datetime-format }
## 仕様更新ルール（Spec-first） { #spec-first }
```

リンク側は相対パス＋アンカー：`[UC-05](./requirements.md#uc-05)`、同一ファイル内は `[§共通 日時フォーマット](#datetime-format)`。

### 既存の明示アンカー一覧（リンク先として実在するもの）

| ファイル | アンカー |
|---------|---------|
| `index.md` | `#spec-first` |
| `requirements.md` | `#learner-extensions`・`#uc-01`〜`#uc-08` |
| `api-spec.md` | `#auth-method`・`#datetime-format`・`#common-error` |

> ASCII 見出し（`### \`GET /api/users/me\`` など）は自動アンカー（`#get-apiusersme`）が効く。api-spec.md 内で実際にこの形で相互参照している（`[\`GET /api/users/me\`](#get-apiusersme)`）。新規参照時は実際に生成されるアンカーを確認すること。

## 各ファイルの記述フォーマット

### api-spec.md — エンドポイント 1 件のテンプレート

エンドポイント追加時は、(1) 該当分類の「エンドポイント一覧」表に行を追加し、(2) 該当 § に以下のブロックを追加する。

````markdown
### `METHOD /api/path` — 概要（権限が限定的なら（ADMIN）等を付す）

#### リクエスト

```http
METHOD /api/path
Authorization: Bearer <JWT>
Content-Type: application/json

{ ...ボディがあれば... }
```

#### クエリパラメータ      ← ある場合のみ

| パラメータ | 型 | 必須 | 説明 |
|------------|-----|------|------|
| `name` | string | ❌ | … |

#### レスポンス（200 OK / 201 Created）

```json
{ ...主要フィールドのみ抜粋... }
```

**XxxResponse 型定義**

| フィールド | 型 | 説明 |
|-----------|-----|------|
| `id` | UUID | … |
````

- リクエストボディがある場合は「**リクエストフィールド（XxxRequest）**」表（列＝`フィールド / 型 / 必須 / バリデーション`）を付ける。
- エンドポイント固有のエラーは末尾に表（`| HTTP | \`code\` | 条件 |`）でまとめる。汎用エラーは [§共通 共通エラーレスポンス](../../../Docs/spec/api-spec.md#common-error) に集約済みなので重複させない。
- 認証・日時・エラー・ページネーションの共通事項は §共通へのリンクで済ませる（各エンドポイントに再掲しない）。

### screen-spec.md — 画面 1 件のテンプレート

画面追加時は、(1) 画面一覧表に行を追加、(2) 画面遷移図（flowchart）にノードと遷移を追加、(3) サイドナビ表に行を追加、(4) 該当 § に以下を追加する。

```markdown
### `/path` — 画面名

**アクセス制御**：（ロール限定画面のみ）ADMIN のみ。他ロールは 403 エラー画面を表示する。

#### UI 要素

| 要素 | 説明 |
|------|------|
| … | …（呼び出す API があれば `GET /api/...` を本文に明記） |

#### バリデーション      ← 入力フォームがある場合のみ

| ルール | 条件 |
|--------|------|
| … | … |
```

### requirements.md — UC・機能要件テーブル

UC を足すときは、(1) 「主要ユースケース一覧」表に `[UC-XX](#uc-xx)` リンク付きで行を追加、(2) 該当 § に UC 見出しと機能要件表を追加する。

```markdown
### UC-09：<ユースケース名> { #uc-09 }

#### 機能要件

| # | 要件 |
|---|------|
| XXX-01 | <要件文> |
```

機能要件の ID は領域接頭辞＋連番（既存：`AUTH-`・`RES-`・`RSV-`・`APRV-`・`USER-`・`DEPT-`・`DASH-`）。新領域なら新接頭辞を定義する。権限を伴う変更は §共通の API 権限マトリクス・画面アクセス権限表にも `✅`/`❌` で反映する。

### er-diagram.md — エンティティ定義表

カラム追加・テーブル追加時は、(1) Mermaid `erDiagram` に型付きで追記、(2) 該当エンティティ定義表に行を追加、(3) 必要ならリレーション説明表・インデックス方針表も更新する。**先に Flyway マイグレーションを書き、それに合わせる**。

```markdown
### <table_name>

| カラム名 | 型 | 制約 | 備考 |
|---------|-----|------|------|
| `id` | `UUID` | PK / NOT NULL | |
| `xxx` | `VARCHAR(100)` | NOT NULL / CHECK (…) | … |
```

- 型・制約は SQL に合わせる（`UUID`・`VARCHAR(n)`・`TIMESTAMP`（`TIMESTAMPTZ` 不可）・`INTEGER`・`BOOLEAN`・`TEXT`）。
- H2 / PostgreSQL 両対応のため PostgreSQL 固有型・関数（`jsonb`・`gen_random_uuid` 等）は使わない（注記済み）。

### enhancements/\<課題\>.md — ビジネス要求シートのテンプレート

エンハンス課題（学習者向け拡張課題）の要件を `Docs/spec/enhancements/<short-desc>.md` に 1 課題 1 ファイルで作成する。配置規約・運用原則は [enhancements/index.md](../../../../Docs/spec/enhancements/index.md) を参照。

```markdown
# <課題名>

> 対象読者：学習者・メンター
> 参照：[../requirements.md](../requirements.md) / [index.md](./index.md)

---

## 背景

<なぜこの課題に取り組むか・どのユースケースを拡張するかを地の文で記述する>

## 要件

| # | 要件 |
|---|------|
| <領域接頭辞>-01 | <要件文> |

## 受入条件

- [ ] <検証可能な完了条件>
- [ ] <検証可能な完了条件>

## 影響範囲

- 対象レイヤー：<frontend / backend / 両方>
- 更新が必要な spec：<requirements.md §xx / api-spec.md §xx / screen-spec.md §xx / er-diagram.md §xx>

## 依存関係

- 前提課題：<完成を前提とする他のエンハンス課題。通常は「なし」>
- 競合する課題：<同一ファイル・テーブル・エンドポイント・マイグレーションを変更する課題と、その共有成果物名。並行着手は非推奨である旨を添える。なければ「なし」>
- 推奨着手順序：<先行・後続が望ましい課題とその理由。なければ「特になし」>

## AI 活用ポイント

<plan mode での計画提示が有効な箇所・AI に検証させやすい観点 等>
```

- 文体・表記は既存 spec ファイルに合わせる（地の文はです・ます調、制約の断定はだ・である調、表は体言止め）。
- 機能要件の ID は `requirements.md` の既存接頭辞（`AUTH-`・`RES-`・`RSV-`・`APRV-`・`USER-`・`DEPT-`・`DASH-`）を流用するか、新規領域なら新接頭辞を定義する。
- 受入条件は**チェックリスト形式**（`- [ ] …`）で、レビュー時に完了条件として個別にチェックできる単位に分割する。
- **依存関係**節は、複数の学習者が課題を並行着手したときのマージ競合を事前に避けるための情報を書く。同一の `ApprovalService` メソッド・同一テーブルへの Flyway マイグレーション（番号衝突）・同一画面コンポーネント等を共有する課題は「競合する課題」に明記する。
- ファイル名は英小文字ケバブケース（`<short-desc>.md`）。他所から参照する明示アンカーが必要な見出しには `{ #id }` を付与する。

## Mermaid 図の使い分け

| 図種 | 用途 | 使用ファイル |
|------|------|-------------|
| `stateDiagram-v2` | ステータス遷移（予約・承認ステップの状態機械） | requirements.md |
| `flowchart` | 画面遷移図 | screen-spec.md |
| `erDiagram` | データモデル | er-diagram.md |
| `sequenceDiagram` | トランザクション的フロー（複数アクター間の往復があるもの。申請→承認、JWT 検証など） | api-spec.md |

> **注意**：単純な取得系 API（一覧・詳細の単発 GET 等）にシーケンス図は不要。アクター間の分岐・副作用・複数往復がある処理に限って描く。

## 相互リンク・参照の記法

- 相対パス＋アンカー：`[requirements.md の UC-05](./requirements.md#uc-05)`
- 同一ファイル内：`[§共通 共通エラーレスポンス](#common-error)`
- `§共通` 参照は「[§共通 認証方式](#auth-method)」のように **§＋セクション名** を本文に書く（節を「§認証」「§リソース」等の慣用名で呼ぶ）。
- 冒頭の `> 参照：` 行に主要な関連ファイルを列挙する（既存全ファイルがこの形式）。

---

## 文体

既存の混在実態に合わせる（無理に統一しない）。

- **説明文（地の文）**：です・ます調（例：「BookFlow の実装仕様を正確に記述します」）。
- **技術的制約・ルールの断定**：だ・である調（例：「全エンドポイントは…付与すること」「アプリ層での排他制御が実装責務」）。
- **表中**：体言止め（例：「リソース別予約一覧・空き確認」「ページ番号（0 始まり）」）。

---

## 注記・補足の記法

- 注意書きは blockquote で `> **注意**：…` 形式：

  ```markdown
  > **注意**：`{stepId}` は `approval_steps.id`（`reservation.id` ではない）。
  ```

- 補足・前提も同じく `>` blockquote（`> **承認者の選定**：…` のように太字ラベルを付けることもある）。

---

## 権限マトリクスの記法

`✅` / `❌` で表す。条件付き許可は丸括弧で補足する。

```markdown
| `GET /api/reservations` | ✅（自分のみ） | ✅（自分のみ） | ✅（全件） |
| `POST /api/resources`   | ❌ | ❌ | ✅ |
```

画面アクセス権限表・サイドナビ表（非表示は `❌（非表示）`）も同じ記法。
