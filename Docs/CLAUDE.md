# CLAUDE.md — Docs 配下 文章規範

`Docs/` 配下の Markdown（仕様書・ガイド・ADR・概要等）を書く・推敲するときに従う、日本語技術文書の文章規範。

対象は文章そのもの（段落構成・論証・語り口・冗長表現）であり、frontmatter や ADR テンプレート形式などの**構造規約**は対象外。構造規約は以下を参照する。

- OKF 準拠 frontmatter：[`Docs/decision/ADR-021-okf-frontmatter-adoption.md`](decision/ADR-021-okf-frontmatter-adoption.md)
- ADR フォーマット：[`Docs/decision/README.md`](decision/README.md)
- Mermaid・ASCII 図の構文規約：[`.claude/rules/aidlc-guardrails.md`](../.claude/rules/aidlc-guardrails.md) §3・§4

## 整形

- 改行は見た目（読みやすさ）を調整するために入れる。改行する箇所は、行末に半角スペースを 2 つ置いて実際に改行されるようにする。スペースを付けない裸の改行は、レンダリング結果ではスペース区切りの地続きの文として表示され、見た目上は何も変わらないため、単独では意味がない。
- **改行のタイミング**は、句点（。）が来るたびに機械的に入れるのではなく、次のいずれかを目安に判断する。
  - **一文・一行が長くなったとき**：目安は 1 行あたり全角 70 文字程度。これを超えて読みづらくなってきたら改行を検討する。
  - **意味のまとまりが区切れたとき**：段落内で話題・視点が転換する境目、または列挙・条件・帰結などの構成単位が完結した位置。
  - 句点ごとに毎回改行すると、かえって幼稚な印象になり流れが悪くなるため避ける。1 文の途中でも文節の区切りで改行してよい。
- 改行を入れてよいのは、通常の段落（地の文）だけとする。次の箇所には適用しない。
  - **箇条書き・番号付きリストの項目内**：改行は「lazy continuation」という不安定な挙動に依存することになり、次の行が別のブロックの開始と誤認される場合がある。1 項目は 1 行のまま書く。
  - **表のセル内**：`|` で区切る表の行は 1 行で完結する必要があり、セル内に改行を入れると行が壊れる。
  - **admonition（`!!! note` 等）の 1 行説明**：改行を入れる場合は、インデントを保ったまま独立した行に分け、インデント幅（4 スペース）を必ず維持する。
- コード、差分、ログ、設定ファイルの断片はコードブロックで示す。
- **`~`（チルダ）・`^`（キャレット）・`==`（等号 2 つ）を地の文で裸のまま書かない**。本リポジトリの Markdown 拡張（`pymdownx.tilde` / `pymdownx.caret` / `pymdownx.mark`）はこれらを下付き・上付き・取り消し線・強調表示として解釈するため、`~/.claude` のようなホームディレクトリパスやバージョン表記、強調目的の `==` は、必ずコードスパン（バッククォート）で囲む。既存文書ではこの原則が徹底されており（例：`` `~/.claude.json` ``）、リライト時にバッククォートを外さないよう注意する。
- 用語の由来や定式化の名称など、本筋から一段外れる補足は、本文に並べず脚注（`[^label]`）に降ろす。脚注ラベルは半角英数字を使う（日本語ラベルでの動作は本リポジトリのビルド環境で未検証のため）。
- 定義や分類の列挙は箇条書きで示してよい。定義される用語は太字にする。
- 用語を本文中で初めて定義・導入するときも、その語は太字にする。すでに導入した語を話題として指すときは「」を使い、太字と使い分ける（初出の定義は太字、以後の言及は「」）。
- ダッシュ（em ダッシュ `—`、horizontal bar `―`、いわゆる2倍ダッシュ「——」）を地の文・見出しで使わない。同格・補足の挿入は括弧（）に、言い換え・敷衍は句点で二文に分けるか読点でつなぐ。範囲を示す en ダッシュ `–` や英語の複合語、コードブロック・書誌情報は対象外。
- 中黒（・）を並列で使わない。ただし単一の固有名詞の内部では使ってよい（例：「サン・マイクロシステムズ」のような中黒を含む固有名詞そのものを指す場合）。
- 見出し・コラム見出しに、区切り線（罫線やダッシュ類）で「種別──主題」のように二要素を詰め込まない。見出しは単一の自然な句にする。
- 用語とその定義を並べる箇条書きは、区切り線ではなく全角コロンで「**用語**：説明」と書く。

## 段落と論証の構成

パラグラフライティングを基本とする。段落は論証の一歩であり、読者は段落単位で論理を追えなければならない。

- 一つの段落には一つのトピックだけを置く。場面の進行（調査、実装、検証、評価）が複数混ざった長い段落は、一歩ずつの段落に分割する。
- 段落の最初の文を読めば、その段落が何の話かわかるようにする。
- 段落の先頭では、前の段落との論理関係を接続表現で明示する（「であれば」「実際」「しかし」）。
- 論証は一方向に進める。結論を出してから反論を処理し、結論を言い直す構成にしない。
- 読者が立てそうな誤った解釈は、明示的に否定してから本当の理由を述べる。
- 「AではなくB」と否定するときは、否定の根拠を一文添える。
- 何かを否定・限定するときは、否定する命題そのものを「」で正確に書き出す。「すべて解決するわけではない」のような漠然とした否定で済ませない。
- 「後の章で扱う」のような前方参照は、論証が一段落した位置（段落末・節末）に置く。

## 論証の厳密さ

文章の論理にツッコミどころを残さない。書き上げたら、読み手の反論を先回りして次の点を点検する。

- 推量・可能性・読者の疑念を表す「かもしれない」「だろう」を、根拠なく機械的に断定へ変えない。断定に直せるのは、本文内の根拠によって命題が確定している場合に限る。
- 異なるものを「同じ」とまとめない。区別すべき対象（別々の決定、別々の原因、種類の違う問題）を一括りの言葉でくくらない。
- 複数の要因がある事象を、単一の原因に還元しない。
- 章・節をまたいで、同じ概念の扱いを一致させる。ある箇所で定義した用語の地位を、別の箇所で変えない。
- 因果を主張するときは、その機構（なぜそうなるのか）を一文で示す。「AだとBになる」とだけ書いて理由を省略しない。
- 検出・保証・解決を「必ず」できるかのように書かない。条件付きで正確に述べる（「〜しやすい」「〜が成り立つときに限り」）。
- 節の中心となる語は、その節以前に定義・対象範囲を述べてから使う。

## 読み手の負荷の管理

読者の記憶と注意は有限の資源として扱う。

- 後で参照する必要のない固有名（ファイル名、関数名、識別子）を出さない。「仕様書」「予約サービス」のような一般的な言い方で済ませる。
- 抽象的な言い回しの指す内容が文脈から一意に決まらないときは、丸括弧による同格挿入でその場で特定し、読者に前を読み返させない。
- 新しい例や場面を追加して読者が保持すべき文脈が増えるときは、前の例と何が違うのか、なぜもう一つ必要なのかを前置きする。
- 導入部で、これから扱う内容に関係しない過剰な詳細を詰め込まない。

## 視点と語り

- 例示では、結果の羅列や受動態（「特定され、判明した」）ではなく、行為者を主語にした動作の連なりで書く。
- 論証の中で読者を「あなた」と呼ばず、役割名（「開発者」「学習者」「メンター」）で書く。
- 対象を指す語は具体的に選ぶ。「AI」「ツール」のような広い語でぼかさない。
- 章や節で術語（ユニットオブワーク、承認ステップ、セルフレビュー等）を導入したら、以後はその語で通す。曖昧語に後退しない。
- 術語・訳語は、その分野で慣用されている語を選ぶ。意味の近い漢語を一般語の感覚で充てない。

## 演出の抑制

演出の規範は全面禁止ではなく、節度の規範である。修辞は、それが効果を生む箇所でのみ使う。

- 本文中の太字強調を多用しない。誤読を防ぐ否定や節の帰結など、論理の要所に限り、一節に一、二箇所まで使ってよい。
- 「〜してはならない」という命令調の断定より、「〜するわけにはいかない」のような、作業者の判断として書く形を選ぶ。
- 「重要なのは〜である」のような前置きで主張を予告しない。主張をそのまま書く。

## LLM っぽい表現の禁止

LLM が大量生成する、中身のない型に誘惑されない。書き上げたら、この節で点検する。

次のような言い回しは、論点を増やさず「ちゃんと書いている感」だけを付ける LLM 口調である。使わない。

- **予告と総括**：「重要なのは〜である」「本節では〜を扱う／探求する」「まとめると」「要するに」（直前の言い換えだけのとき）
- **正面から系**：「正面から扱う」「正面から見る／書く」——中身の代わりに姿勢だけを宣言する
- **空虚な形容**：「不可欠」「核心的」「鍵となる」「根本的な」（主張の中身を説明せず強調だけする）、「多角的」「包括的」「総合的」
- **空虚な動詞**：「掘り下げる」「深掘りする」「言語化する」（何をどう書いたかを示さず終わる）
- **接続の型**：「〜において」「〜という側面から」「〜の観点から」（新情報なし）、「さらに」「また」「加えて」の連打
- **弱い緩和と称賛**：「〜と言えるだろう」（根拠なく主張を弱める場合のみ削る。推量・仮定・読者の疑念なら残す）、「非常に」「極めて」「大いに」

悪い例：「本節では、承認フローの仕組みを正面から扱う」「重要なのは、多角的な観点から見た整合性である」。
良い例：「本節では、承認フローの仕組みを扱う」「整合性は、承認者が誰かを一意に決められるかにある」。

## 冗長の排除

無駄な文章をなるべく残さない。

- 同じ主張を言い換えて繰り返さない。一つの主張は一度だけ書く。
- 隣接する節が同じことを別の角度で述べているなら、役割が重複している。片方に吸収して一つの節にまとめる。
- 場面を描写した直後に、その内容を要約し直さない。
- 同じ論理的役割を持つ並列の事実は、文を分けて重ねず一文にまとめる。
- 読者が自力で補える中間段階の説明は書かない。
- 接続や評価のためだけの文（「それ自体はよいことである」など）を置かない。
- 本文でまだ導入していない概念や文書名を、先回りして持ち出さない。

## 見出しの付け方

見出しは内容を特定できる具体的なものにする。その節が答える問い、または扱う対象を指す句にする。

- 作業の手順だけを述べる見出し（「設定を直す」等）や、情報量のない見出しにしない。
- 見出しを、節の結論を言い切る「セリフ」にしない。見出しの時点で読者がオチを知る状態を避ける。
- 節で扱っている対象を指す名詞句でもよい（例：「承認ステップの状態遷移」）。

---

## 読者への誠実さ

- 例が作為的に見えうる場合、それを隠さない。読者の疑念を先回りして認め、現実に十分あり得ることの根拠を短く添える。
- 確認していないことを、確認したかのように滑らかに書かない。
