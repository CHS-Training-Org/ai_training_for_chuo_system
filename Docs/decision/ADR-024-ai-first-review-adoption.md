# ADR-024 — AI一次レビュー（検討A）の採用

## Status

Accepted（2026-07-20）

## Context

[ADR-023](./ADR-023-mentor-gate-removal.md) により、BookFlowは学習者のセルフレビュー・セルフマージによる自己完結運用へ移行した。ADR-023のConsequencesは「メンターによる第三者レビューが保証されなくなるため、コードの誤りが早期に発見されない可能性がある」ことを今後の課題としていた。

この課題への対応として、operations-guide.mdおよび[review-criteria.md](../guide/review-criteria.md)の前方互換メモは「検討A（AI一次レビュー）」「検討A案2（AIレビュー基盤）」を未決事項として残していた（本ADRの決定に伴い、両メモは決着済みの内容へ更新した）。検討A案2は「追加E（リポジトリの公開/非公開の確定）」を前提としており、GHAS・CodeQL等の設定変更を伴う想定だった。

検討Aの結論を出すにあたり、次の3点を検討した。

**トリガー方式**：当初はPR作成をトリガーに自動レビューする構成、またはPRのレビュアー欄にClaudeを指定する構成を検討した。前者はPR作成・push毎にActions実行コストが積み上がる。後者はGitHubの仕様上実現できない。GitHub Copilotの`copilot-pull-request-reviewer[bot]`はGitHub側が用意した専用機能であり、Claude側（`anthropics/claude-code-action`）に同等のレビュアー選択肢は存在しない。ラベル付与によるオプトインも検討したが、PR上でのやり取りは複数回発生しうるため、初回だけラベルで起動しても以降のやり取りには結局コメントが必要になり、学習者を混乱させると判断した。

**レビュー内容**：BookFlowはAI-DLCエンジンを通してエンハンス課題を進め、仕様・テスト・ドキュメント・コードを同一のAI-DLC実行でまとめて生成し、同一PRでコミットする。この開発モデルは、人間が段階的に書いたコードを他者がレビューする一般的なPRレビューとは前提が異なるため、レビュー観点を一般的なAIコードレビュー（バグ・セキュリティ・スタイルの検出）から独自に組み直す必要があった。一般的な観点は既存の`code-review`公式プラグインやCIのlint/testと役割が重複する。

**リポジトリの公開設定**：本リポジトリは現在Publicであり、検討A案2が前提としていた「追加E」は確定した。ただし確定後もGHAS・CodeQLの採用は学習用スコープ外と判断した。

## Decision

検討Aを**採用**する。ただし当初想定していた構成（PR作成トリガー・レビュアー指定トリガー・一般的なAIコードレビュー観点・GHAS/CodeQL）はいずれも採用しない。

- **トリガー**：`@claude pr-review`というコメントをPRに投稿したときにのみ起動する。PR作成時の自動起動、レビュアー指定による起動は採用しない。
- **レビュー内容**：以下の3観点で構成する。いずれもBookFlowの開発モデル（AI-DLCによる仕様・テスト・ドキュメント・コードの一括生成、セルフマージ運用）に対応させた観点であり、一般的なバグ検出とは役割が異なる。
    - **要求整合性チェック**：対応するビジネス要求シート（`Docs/spec/enhancements/`）の要件表・受入条件と、PRの差分を比較する。シートがPR本文にリンクされていない場合はこのチェックを省略する。
    - **実装と非機能部分の整合性チェック**：コード変更に対応するテストコードの変更が存在し、新しい振る舞いを実際に検証しているか、仕様・ADRの前提が変わった場合にドキュメント更新が同一PRに含まれているかを確認する。
    - **理解度チェックのための質問**：PRの変更内容について学習者が自分で説明できるはずの設計判断・境界値・エラーハンドリングを2〜4問、質問として投稿する。回答は強制せず、ブロッキングもしない。
- **静的解析（GHAS/CodeQL）は不採用**とする。学習用スコープ外と判断し、検討A案2が前提としていたGHAS/CodeQL導入は行わない。両ガイドの前方互換メモにあるGHAS・CodeQLの記載は削除する。
- **非ブロッキング**：レビューはPRコメントとして投稿するのみで、必須status checkには加えない。ADR-023のセルフ完結運用を維持する。

実装は`.github/workflows/claude.yml`に既存の`claude`ジョブ（汎用`@claude`メンション）とは別の`claude-review`ジョブを追加する形で行う。`claude-review`ジョブの権限は`contents: read`・`pull-requests: write`のみとし、コード変更・コミットを行わせない。

観点の本文は`.github/workflows/references/pr-review-rubric.md`に切り出し、`prompt`にはこのファイルを読んで従うよう指示するのみとする。観点の改訂をワークフローYAMLの変更から分離するためである。このファイルは`main`から読み込まれる（`actions/checkout`の仕様上、レビュー対象PRの差分ではなく`main`がチェックアウトされる）ため、レビュー対象のPR側でこの内容を書き換えて判定を誘導することはできない。`Docs/`配下ではなく`.github/`配下に置くのは、Zensicalのドキュメントサイト（`docs_dir = "Docs"`）のビルド対象に含めず、学習者の目に意図せず触れないようにするためである。

## Consequences

**ポジティブ**：
- 学習者は理解度チェックの質問に答えられない箇所があれば、AI生成コードを無検証でコミットしていたことを自覚できる。これは[ai-tools-guide.md](../guide/ai-tools-guide.md)が明記する「生成コードの責任はマージした開発者が負う」という方針を、外部からの問いによって検証可能にする。
- `@claude pr-review`と明示的にコメントした時のみ実行されるため、Actions実行コストが無制限に積み上がらない。
- 要求整合性・実装と非機能部分の整合性チェックは、CIが検出できない「要件の実装漏れ」「新規ロジックに対するテストの欠落」を補う。

**留意点**：
- フォークPRでは`secrets`が渡らず`claude-review`は動作しない。BookFlowは同一リポジトリのfeatureブランチで開発する運用（`feature/<GitHubユーザー名>/<issue番号>-<short-desc>`）が前提のため、実運用上は問題にならない。
- レビューはコメント投稿のみで、レビュー内容への対応を強制する仕組みはない。ADR-023同様、学習者の自発性に依存する。
