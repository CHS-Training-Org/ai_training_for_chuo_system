## 学習者ペルソナと到達目標

### ペルソナ A：新人エンジニア

- **前提スキル**：プログラミング基礎（いずれかの言語）、Git 基本操作
- **AI ツール経験**：ほぼなし
- **到達目標**：
  - DevContainer を自力で起動し、ベースサービスの動作を確認できる
  - AI ツールを補助として使いながら、既存コードを読み解いて小規模な機能追加ができる
  - PR を作成し、メンターのレビューを受けてマージできる

### ペルソナ B：中堅エンジニア

- **前提スキル**：実務開発経験（どの言語・FW でも可）、コードレビュー経験
- **AI ツール経験**：断片的に使ったことがある程度
- **到達目標**：
  - AI ツールを使い分けて独立した機能設計〜実装〜テストを完結できる
  - 繰り返し予約・帳票出力・多段階承認など複雑な拡張課題を自己完結できる
  - 他の学習者の PR に対してレビューコメントを書ける

## 必須ステップ課題（順序性あり）

GitHub Issue に `required` + `sequential` ラベルを付与して登録する。

### STEP-01：開発環境のセットアップ

**ゴール**：DevContainer を起動し、ブラウザで BookFlow のダッシュボードにアクセスできる状態を作る

| 項目 | 内容 |
|------|------|
| 推奨レベル | Beginner |
| 推定工数 | 1〜2 時間 |
| AI 活用例 | トラブルシュート時に Copilot Chat で `docker: error response` などエラーメッセージを質問する |
| 完了条件 | `http://localhost:3000` でトップページが表示される、`http://localhost:8080/actuator/health` が `200 OK` を返す |

**手順（Issue 本文に記載する内容）**：
1. リポジトリをクローン
2. VS Code で開き、Dev Containers 拡張で「Reopen in Container」
3. コンテナ起動後、`pnpm dev`（frontend）と `./gradlew bootRun`（backend）の実行を確認
4. `scripts/seed.sql` を実行して初期データを投入

### STEP-02：コードベース把握

**ゴール**：BookFlow のコードを読んで、フロントエンド→BFF→バックエンド→DBの流れを説明できる

| 項目 | 内容 |
|------|------|
| 推奨レベル | Beginner |
| 推定工数 | 2〜3 時間 |
| AI 活用例 | Claude Code CLI で `"リソース一覧の処理フローを説明して"` と聞いて、コードを読み解く |
| 完了条件 | リソース一覧表示のリクエストが、Next.js → Server Actions → Spring Boot → PostgreSQL を辿る流れをコメント付きで図示できる |

### STEP-03：AI ツールの導入と使い分け

**ゴール**：GitHub Copilot と Claude Code CLI それぞれの特性を理解し、意図的に使い分けられる

| 項目 | 内容 |
|------|------|
| 推奨レベル | Beginner |
| 推定工数 | 1〜2 時間 |
| AI 活用例 | `Docs/guide/ai-tools-guide.md` を参照。両ツールで同じ質問をして回答を比較する |
| 完了条件 | `Docs/guide/ai-tools-guide.md` の「使い分けチェックリスト」に回答を記入してPRを提出 |

### STEP-04：既存機能の読解（テスト観点）

**ゴール**：予約申請機能のユニットテストを読み、テストの意図と網羅範囲を説明できる

| 項目 | 内容 |
|------|------|
| 推奨レベル | Beginner |
| 推定工数 | 1〜2 時間 |
| AI 活用例 | Copilot Chat で `"このテストが検証していない境界値は？"` と確認する |
| 完了条件 | 予約申請のサービス層テストに「テスト意図」のコメントを日本語で追記してPRを提出 |

### STEP-05：小規模機能改修

**ゴール**：リソース一覧の表示順を「登録順」から「カテゴリ別」に切り替えるフィルター機能を追加する

| 項目 | 内容 |
|------|------|
| 推奨レベル | Beginner |
| 推定工数 | 3〜5 時間 |
| AI 活用例 | Copilot でクエリ生成、Claude Code CLI で「他の箇所に影響がないか確認して」と依頼する |
| 完了条件 | フロントエンドのセレクトボックスで切替ができ、バックエンドのユニットテストが通る |

## 選択課題（順序性なし）

GitHub Issue に `optional` ラベルを付与して登録する。学習者が自由に選択して取り組む。

### Level: Beginner

| 課題タイトル | 対象レイヤー | 推定工数 |
|------------|------------|---------|
| リソース詳細画面に場所・定員情報を追加表示 | frontend | 2〜3h |
| リソース一覧のカテゴリフィルター改善 | frontend | 3〜5h |
| 予約申請前の確認ダイアログ追加 | frontend | 2〜3h |
| キャンセル済み予約の一覧表示改善 | frontend + backend | 3〜5h |
| バックエンドのユニットテストを追加する | backend | 3〜5h |

### Level: Intermediate

| 課題タイトル | 対象レイヤー | 推定工数 |
|------------|------------|---------|
| 繰り返し予約機能（毎週・毎月）の追加 | frontend + backend | 6〜10h |
| 予約重複時の代替時間提案 | backend | 5〜8h |
| 利用実績の集計・グラフ表示（月次） | frontend + backend | 5〜8h |
| 部署ごとの承認者設定（管理者機能） | frontend + backend | 6〜10h |
| 承認コメントのスレッド表示 | frontend + backend | 4〜6h |

### Level: Advanced

| 課題タイトル | 対象レイヤー | 推定工数 |
|------------|------------|---------|
| カレンダービュー（週/月表示）の実装 | frontend | 8〜12h |
| CSV / PDF 帳票出力（予約実績レポート）| frontend + backend | 6〜10h |
| 多段階承認フローの設定機能 | frontend + backend | 8〜12h |

## AI 駆動開発 活用ガイドライン

### GitHub Copilot（VSCode 拡張）を使う場面

- コードの自動補完・インライン提案（タイピング中）
- 既存パターンと同様のコードを繰り返す際の効率化
- Copilot Chat でエラーメッセージや API の使い方を質問

### Claude Code (CLI) を使う場面

- 「このファイル全体を読んで設計を理解して」など、広いコンテキストが必要な質問
- 複数ファイルをまたぐリファクタリング案の検討
- 「この変更が他の箇所に影響しないか確認して」など、安全性の確認
- テストケース・ドキュメントの生成

### 禁止事項・注意事項

- 社内機密情報（本番 DB 接続情報・APIキー等）を AI に入力しない
- AI が生成したコードをそのままコミットしない。必ず意図を理解してからマージする
- AI の回答が正しいとは限らない。`Docs/ARCHITECTURE.md` や公式ドキュメントで必ず確認する
- AI 生成コードの責任は、それをレビュー・マージした開発者が持つ

## 評価基準（メンター向け）

選択課題・必須課題の完了判定は「完了条件チェックリスト全通過」を基本とする。

コードレビューでは以下の観点を確認する（学習段階に応じて重みを調整）：

| 観点 | 新人重点 | 中堅重点 |
|------|---------|---------|
| 動作確認（完了条件通過）| ◎ | ◎ |
| コードの可読性 | ○ | ◎ |
| 既存パターンへの整合性 | △ | ○ |
| テストの妥当性 | △ | ○ |
| AI 活用の適切さ（盲目的な貼り付けがない）| ○ | ◎ |
| 設計の意図説明（PR 本文）| ○ | ◎ |
