## 役割と責務

| 役割 | 人数目安 | 主な責務 |
|------|---------|---------|
| **リポジトリオーナー** | 1名 | 設定管理・マイルストーン承認・最終レビュー |
| **メンター** | 2〜4名 | 課題設計・PR レビュー・質問対応・課題メンテナンス |
| **学習者** | 制限なし | 課題実施・PR 提出・フィードバック提供 |

---

## 質問・サポートフロー

### 推奨チャネル

1. **GitHub Issue のコメント**：課題そのものに対する質問はその Issue のコメントに投稿する（他の学習者も参照できるため優先）
2. **PR コメント**：実装についての具体的な質問は PR 上でメンターに @mention する
3. **社内チャット（Slack 等）**：緊急のトラブル・環境構築の重大障害のみ使用

### メンターの SLA

| 種別 | 目標応答時間 |
|------|------------|
| 環境構築・起動に関する質問 | 翌営業日以内 |
| PR レビュー | 3 営業日以内 |
| 課題内容の質問 | 翌営業日以内 |

## PR・Issue 運用フロー

### 学習者側フロー

```
1. 取り組む Issue を選ぶ
   └── `in-progress` ラベルを自分で付ける（二重取り防止）

2. main から feature ブランチを作成
   └── `feature/<issue番号>-<短い説明>`

3. 実装・コミット（Conventional Commits）

4. PR 作成
   └── PR テンプレートに従い、実装意図・AI 活用内容・動作確認結果を記載

5. メンターのレビューを受ける
   └── 修正 → 再レビュー のサイクル

6. Approve 後、学習者または メンターが main にマージ

7. Issue をクローズ
```

### PR テンプレートに含める項目

- 対応 Issue 番号（`Close #XX`）
- 実装内容の概要（3行以内）
- AI ツールを使った箇所とその内容
- 動作確認手順と結果
- 特にレビューしてほしいポイント

## 課題（Issue）のライフサイクル

### 課題の追加

- メンターが Issue テンプレートに従って登録
- ラベルを正確に付与（level / type / required|optional / sequential）
- 追加タイミング：Phase 3（課題登録フェーズ）に一括登録、以降は随時追加可

### 課題の更新・修正

- 手順が変わった・難易度が変わったなど、学習者からフィードバックがあった場合に更新
- 大幅な変更の場合は、進行中の学習者への影響をコメントで通知する

### 課題の廃止

- 依存ライブラリの互換性がなくなった、またはベースサービスの変更で課題の前提が崩れた場合
- `deprecated` ラベルを付与してクローズ。理由を Issue 内にコメントする

## リポジトリ維持

### 依存更新ポリシー

| 対象 | 頻度 | 担当 |
|------|------|------|
| `frontend/` npm 依存（pnpm update）| 月次 | メンター or Dependabot |
| `backend/` Maven 依存（pom.xml）| 月次 | メンター |
| ベースイメージ（Docker）| 四半期 | リポジトリオーナー |
| Java / Node.js メジャーバージョン | 年次（LTS 確認後）| リポジトリオーナー |

Dependabot を有効化し、PR を自動作成する。マージ前に CI が通ることを確認する。

### セキュリティ対応

- `security-scan.yml`（GitHub Actions）が週次で依存スキャンを実行
- Critical / High の脆弱性は 1 週間以内に対応
- Secret（APIキー等）が誤ってコミットされた場合は即座にオーナーへ報告し、キーをローテーション

## AI ツール運用ポリシー

### 利用範囲

| 用途 | 許可 | 注意点 |
|------|------|--------|
| コード補完・提案 | ○ | 内容を理解してから採用する |
| コード解説・質問 | ○ | 公式ドキュメントと照合する |
| テスト・ドキュメント生成 | ○ | レビューを必ず通す |
| リファクタリング提案 | ○ | 挙動が変わらないことをテストで確認する |
| 本番データの入力 | ✗ | 絶対禁止 |
| 社内機密情報の入力 | ✗ | 絶対禁止 |

### 生成コードの責任

- AI が生成したコードをマージした開発者が、そのコードの品質・安全性に対して責任を持つ
- PR 本文に「AI ツールを使った箇所」を明記することを必須とする

### GitHub Copilot の管理

- Copilot Business ライセンスが必要。組織管理者が有効化する
- DevContainer 内の VS Code 設定で `GitHub.copilot` 拡張を自動インストールする

### Claude Code (CLI) の管理

- 個人の Anthropic / Claude.ai アカウントで利用する
- `CLAUDE.md` をリポジトリルートに配置し、プロジェクトコンテキストを提供する

## CI/CD・品質ゲート

### GitHub Actions ワークフロー

| ワークフロー | トリガー | 内容 |
|------------|--------|------|
| `ci-frontend.yml` | PR / main push | ESLint + Prettier チェック、Vitest ユニットテスト |
| `ci-backend.yml` | PR / main push | Checkstyle + SpotBugs、Maven テスト（`./mvnw test`）|
| `security-scan.yml` | 週次・PR | npm audit、OWASP Dependency-Check |

### 品質ゲート（PR マージ要件）

- [ ] CI ワークフローが全て Green
- [ ] 1名以上の Approve（メンターまたはオーナー）
- [ ] 未解決のコメントが 0 件

## 学習効果測定

### 定量指標（Issue ラベルから集計）

- 課題の完了数・レベル別完了分布
- PR のサイクルタイム（Issue クローズから PR マージまでの時間）

### 定性指標（アンケート）

学習終了後に以下を Google Forms 等で収集する（任意回答）：

- 環境構築のしやすさ（1〜5）
- 課題の難易度感（難しすぎ / 適切 / 簡単すぎ）
- AI ツールは役に立ったか（1〜5）
- 改善してほしい点（自由記述）

### フィードバックの反映

- 四半期ごとにメンター会議でフィードバックをレビュー
- 課題の難易度調整・新課題追加・ドキュメント改善に反映する
