## 命名規則・フォーマット

### ファイル命名

- `plan/`：`NN_SNAKE_CASE.md`（NN は 2桁の連番）
- `spec/`：`kebab-case.md`
- `guide/`：`kebab-case.md`
- `decision/`：`NNN_kebab-case.md`（NNN は 3桁の連番）

### 図のフォーマット

- **Mermaid を推奨**（GitHub / VSCode で描画確認可能）
- 複雑な図は draw.io（`.drawio.png` 形式でリポジトリに含める）
- PlantUML は採用しない（ローカル環境依存が大きいため）

### ドキュメントのヘッダー形式

全ドキュメントの冒頭に以下を記載する：

```markdown
> 対象読者：xxx
> 参照：[ファイル名](../path)
```

### 更新履歴

git の履歴をそのまま変更履歴として使用する。ドキュメント内に手動の更新履歴セクションは設けない。

## ドキュメントの作成・更新フロー

```
1. 変更内容を決める
   └── 大きな変更は Issue を立ててから作業する

2. feature ブランチを作成して編集

3. PR を提出
   └── 対応 Issue を記載、変更意図を説明

4. メンターまたはオーナーがレビュー・Approve

5. main にマージ
```

### 学習者によるドキュメント修正

学習者がガイドのミスや情報不足を発見した場合、PR で修正を提案できる。`type:docs` ラベルの Issue を作成して報告することも歓迎する。

## `ai-tools-guide.md` の必須記載内容

学習者が最も参照するガイドのため、記載事項を定義する。

1. **GitHub Copilot の使い方**
   - VSCode でのインライン補完の受け方（Tab キー）
   - Copilot Chat の開き方と効果的な質問の仕方

2. **Claude Code (CLI) の使い方**
   - インストール方法と初期設定（`claude` コマンド）
   - DevContainer 内での使い方
   - 効果的なプロンプトの書き方（コンテキストを先に伝える）

3. **使い分けチェックリスト**（STEP-03 の完了条件で使用）
   - 「今どちらを使うべきか」の判断フローチャート

4. **禁止事項**（`05_OPERATION_GUIDE.md` からの抜粋）

## `getting-started.md` の必須記載内容

STEP-01 の手順書として機能するため、以下を漏らさず記載する。

   - VS Code
   - Dev Containers 拡張
