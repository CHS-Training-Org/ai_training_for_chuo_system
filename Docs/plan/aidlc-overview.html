0. この資料について

カテゴリ3（AI 駆動開発ワークフローの整備）は、3.1〜3.6 ですでに AI-DLC を「思考モデル」として BookFlow に写像済みです （Docs/guide/dev-workflow.md・CLAUDE.md・ビジネス要求シート・Issue/PR テンプレート・CI ゲート）。 3.7 はその先で、AWS Labs 公式リポジトリ awslabs/aidlc-workflows の実ファイル（ワークフロールール定義）を導入するかどうかを問うタスクです。

読み方

§1 で AI-DLC（公式）が何を定義しているかを把握し、§2 で BookFlow が現在どこまでをすでに実現しているかを確認します。 §3 では公式をそのまま入れた場合にどこがぶつかるかを整理し、§4 でその衝突を踏まえた組み合わせ方の選択肢（中立に3案）を提示します。 どの案を選ぶか・あるいは選ばないかは、この資料を読んだうえでのメンター判断に委ねます。

1. AI-DLC とは

1.1 方法論の狙い

AI-DLC（AI Development Life Cycle）は AWS Labs が公開している、AI コーディングエージェント向けの開発方法論（ワークフロールール集）です。 ソフトウェア開発を「要件定義（Inception）→ 設計・実装（Construction）→ 運用（Operations）」の3フェーズに分け、 各フェーズの中で AI エージェントが取るべき手順・生成すべき成果物・人間に確認すべきタイミングを細かく規定します。 中心となる考え方は次の2つです。

Plan-First（計画提示→承認）

AI はいきなりコードを書かず、各段階でまず「計画（チェックボックス付きの実施内容リスト）」を提示し、 人間が明示的に承認するまで次の段階に進みません。

Units of Work（作業単位への分解）

大きな実装は「ユニット」（マイクロサービス・モジュール・ビジネストランザクションなどの単位）に分解し、 ユニットごとに設計→コード生成のループを回します。

1.2 3フェーズと plan-first ゲート

公式の core-workflow.md は次のフロー全体を1ファイルで規定しており、冒頭に "This workflow OVERRIDES all other built-in workflows"（このワークフローは他の組み込みワークフローすべてを上書きする） と明記されています。

1.3 units of work・Two-Level Checkbox・audit.md

Units of Work の分解

Workflow Planning の段階で、要件が複数サービス・複数モジュールにまたがると判断された場合に Units Generation を実行し、 マイクロサービス単位・モジュール単位・ビジネストランザクション単位などでユニットに分解します。 各ユニットは Construction フェーズで個別に設計〜コード生成のループを回します。

Two-Level Checkbox Tracking

段階内（Plan-Level）と全体（Stage-Level）の2階層でチェックボックスによる進捗管理を行います。

# レベル1：Plan-Level（段階内の詳細進捗。各計画ファイル内）
## Code Generation Plan - Unit: auth-service
- [x] API エンドポイント定義
- [x] ドメインモデル設計
- [ ] ビジネスロジック実装
- [ ] ユニットテスト作成

# レベル2：Stage-Level（全体進捗。aidlc-docs/aidlc-state.md）
## Workflow Progress
- [x] Workspace Detection
- [x] Reverse Engineering (skipped - greenfield)
- [x] Requirements Analysis
- [ ] Code Generation (Unit: auth-service)

「完了したら同一インタラクション内で即座にチェックボックスを更新する」ことが MANDATORY（必須） ルールとして定義されています。

audit.md（操作ログ）

すべてのユーザー入力と AI のアクションを、ISO 8601 タイムスタンプ付きで aidlc-docs/audit.md に追記し続けます。 「ユーザー入力は要約せず全文を記録する」「ファイル全体の上書きは禁止（追記のみ）」という制約があります。 

## Requirements Analysis
**Timestamp**: 2024-01-15T14:32:00Z
**User Input**: "[ユーザーからの完全な入力 - 要約禁止]"
**AI Response**: "[実行したアクションまたは応答内容]"
**Context**: Requirements深度=Standard, 機能スコープ確認中

3.2 主要衝突の詳細

① 並行ドキュメントツリー（aidlc-docs/）

公式ワークフローは実行されるたびに aidlc-docs/inception/・aidlc-docs/construction/<unit>/ 以下に 要件分析・設計・コード要約などの成果物を生成します。BookFlow ではこれに相当する内容を ビジネス要求シート（Docs/spec/enhancements/）・仕様（Docs/spec/）・実装そのものに直接反映する運用です。 公式ワークフローを「実行」すると、本来 Docs/spec/enhancements/ に書くべき内容が aidlc-docs/ にも生成され、 二重管理（並行ドキュメントストア）が発生します。

② CLAUDE.md の上書き

公式の Claude Code 向けインストール手順は core-workflow.md を ./CLAUDE.md としてコピーする前提です。 本リポジトリの CLAUDE.md は技術スタック・ディレクトリ構成・コマンド・コーディング規約・3.5で追記した「AI 駆動開発の進め方」など BookFlow 固有の情報を持つため、単純な上書きは成立しません。

③ マルチエージェント対応ファイルの混入

公式 zip には Cursor・Amazon Q・Kiro・Cline・Copilot 向けの配置物が含まれます。BookFlow は「AGENTS.md は導入しない・Claude Code 専一」 と決定済み（タスク2.2）であるため、これらは導入対象から明確に除外する必要があります。

④ 規模・粒度（全件導入の要否）

公式 rule-details は過信防止・コンテンツ検証・質問フォーマット・深度レベル・reverse-engineering など約25ファイルの細則からなる 本格的な方法論です。カテゴリ3冒頭は「成果物の出力先・対象エージェントをBookFlowの既存資産に合わせてカスタマイズして適用する。 plan-firstゲート・Spec-first・セルフレビューといったプロセスの厳格さは簡略化しない」と定めており、全件をそのまま vendoring する必要はない一方、採用する部分についてはプロセスの厳格さを妥協しません。3.7ではこの方針に基づき、思考ガードレール系 common/ 全11ファイルを選別して vendoring し、採用台帳で反映状況を管理しています。

⑤ "OVERRIDES all built-in workflows" の宣言

core-workflow.md 自身が「他の組み込みワークフローすべてを上書きする」と宣言しているため、 そのまま導入すると Claude Code の plan mode の使い方や既存スキル（/update-spec・/draft-pr）との関係が 未定義になります。3.1〜3.6 で構築した導線（plan mode → 第1ゲート → Spec-first → ... → 第2ゲート）と、 公式ワークフローの内部フロー（Inception → plan-firstゲート → Construction の per-unit ループ）のどちらが「主」になるのかを 整理しないまま導入すると、学習者が2つのワークフロー定義の間で混乱する可能性があります。

3.4 verbatim full install が成立しない理由

以上を踏まえると、公式の手順どおり「core-workflow.md を CLAUDE.md として配置し、 aidlc-rule-details/ を丸ごとコピーする」という verbatim（そのまま）の full install は、 少なくとも次の3点で BookFlow の確立済み方針と直接矛盾するため、選択肢として成立しません： (1) 既存 CLAUDE.md の上書きになる、(2) aidlc-docs/ という並行ツリーを作ってしまう、 (3) 「簡素化」原則に反する規模・複雑度を持ち込み、かつ "OVERRIDES" 宣言により既存導線との関係が未定義になる。 したがって §4 の取り込み案は、いずれも「公式ファイルの全部または一部を、BookFlow の既存方針に適合する形に 変換・選別したうえで取り込む（あるいは取り込まない）」という枠組みの中で検討します。

4. 取り込み案

以下の3案を中立に並べます。優先順位や推奨は付けていません。3.7 のタスク記述にある 「着手可否は既存の『写像する』方針との整合性をメンターが評価したうえで判断する」を踏まえ、 どの案を選ぶか（あるいはどれも選ばないか）はメンターの判断に委ねます。

案A：最小サブセット参照＋薄いアダプタ

公式 rule-details のうち、3.1〜3.6 でまだ反映されていない部分 （overconfidence-prevention.md・content-validation.md・depth-levels.md・question-format-guide.md・reverse-engineering.md など） をキュレートして選別し、出典（GitHub URLと VERSION）を明記した上で .claude/ 配下に参照配置する。 これらの内容を、既存資産（Docs/spec/enhancements/・/update-spec・/draft-pr・dev-workflow.md）に 向けて読み替える「アダプタ」文書を作成し、CLAUDE.md には1行のポインタのみ追加する。

観点内容
追加されるファイル.claude/ 配下に数ファイル（選別した rule-details の写し＋出典）＋アダプタ文書1〜2点
CLAUDE.md への影響1行のポインタ追加のみ（上書きなし）
aidlc-docs/ の扱い作らない（既存 Docs/spec/・Docs/guide/ に向けてアダプタが読み替える）
長所過信防止・質問形式・reverse-engineering など、3.1〜3.6で抜けていた実用的なガードレールを取り込める。出典が明確で provenance を保てる
短所選別・読み替えの作業量が発生する。公式の更新（VERSION 変更）への追従が今後の運用課題になる
想定作業量中（選別5ファイル程度＋アダプタ文書の作成・既存ガイドへの相互リンク）

案B：ポリシー文書化のみ

公式ファイルの vendoring（コピー）は行わない。3.1〜3.6 の写像で未反映のデルタ（過信防止・質問形式・深度レベルなど）のうち、 BookFlowに適用する価値があるものを、Docs/guide/dev-workflow.md や CLAUDE.md に BookFlow独自の文章として要約・記述する。公式リポジトリへは出典リンクのみで参照する。

観点内容
追加されるファイル新規ファイルなし。既存 dev-workflow.md / CLAUDE.md への追記のみ
CLAUDE.md への影響既存セクションへの追記（上書きなし）
aidlc-docs/ の扱い作らない（そもそも公式ファイルを導入しないため発生しない）
長所リポジトリにファイルを増やさない。最も「簡素化」原則に忠実。メンテナンス対象が増えない
短所公式ルールの細かいニュアンス（質問ファイル形式の具体的書式など）が失われやすい。公式更新との対応関係が薄い
想定作業量小（既存2ファイルへの追記）

案C：非採用を記録

3.1〜3.6 の写像により AI-DLC の思考モデルは実質的にすでに BookFlow に組み込まれていると判断し、 3.7 を「公式ワークフローファイルの導入は行わない」という結論として Docs/plan/PHASE4_AI_DRIVEN_DEV_TASKS.md に記録してクローズする。

観点内容
追加されるファイルなし（計画書のステータス更新のみ）
CLAUDE.md への影響なし
aidlc-docs/ の扱い該当なし
長所作業ゼロ。既存の「並行ツリーを作らない」方針を最も強く維持できる
短所§1.5 で挙げた思考ガードレール（過信防止・質問形式・reverse-engineering 等）は一切取り込まれない
想定作業量極小（計画書のステータス変更とメモ記載のみ）
