# AI ツール活用ガイド（Claude Code）

このガイドは STEP-03「AI ツール導入・活用」の参照資料です。本リポジトリの標準 AI ツールである **Claude Code** の特性を理解し、状況に応じて意図的に活用できるようになることを目標とします。

STEP-03 の完了条件：このガイドの「[活用チェックリスト](#checklist)」セクションに回答を記入して PR を提出してください。

---

## Claude Code とは

Claude Code は、ターミナル上で動作するコーディングエージェントです。エディタのインライン補完と異なり、**リポジトリ全体を読み取りながら対話形式でタスクを進められる**のが特徴です。

得意な場面：

| 場面 | 例 |
|------|----|
| 広いコンテキストが必要な質問 | 「予約申請から承認までの処理がどのファイルを通るか説明して」 |
| 複数ファイル横断の変更 | 「`Resource` に `description` フィールドを追加して。Entity・DTO・画面まで一式」 |
| 影響範囲の安全確認 | 「このメソッドのシグネチャを変えたら、呼び出し元はどこが壊れる？」 |
| テスト・ドキュメント生成 | 「この Service のユニットテストを既存テストの書き方に合わせて作成して」 |
| エラー原因の調査 | エラーメッセージを貼り付けて「原因を調査して」 |

## セットアップ { #setup }

Claude Code は **DevContainer に組み込み済み**です。学習者が追加でインストールする必要はありません。

| 構成要素 | 内容 |
|---------|------|
| CLI | devcontainer feature（`anthropics/devcontainer-features/claude-code`）で自動インストール |
| VS Code 拡張 | `Anthropic.claude-code` が自動インストール |
| 認証情報 | ホストの `~/.claude` / `~/.claude.json` をコンテナへバインドマウント |

### 初回の認証

DevContainer 内のターミナルで起動します。

```bash
claude --version   # インストール確認
claude             # 対話セッションを開始
```

初回起動時はログイン（ブラウザ認証）を求められます。画面の指示に従って認証してください。認証情報はマウントされたホスト側の `~/.claude.json` に保存されるため、**コンテナを再作成してもログインし直す必要はありません**。

!!! tip "ホストで Claude Code を使ったことがある場合"
    ホスト側の認証情報がそのままコンテナに共有されるため、ログイン不要で使い始められます。

!!! warning "コンテナ起動が `.claude.json` 関連で失敗する場合"
    マウント元の `~/.claude.json` はコンテナ初回起動時に自動生成されます（`devcontainer.json` の `initializeCommand`）。
    起動に失敗する場合は [troubleshooting.md §AI ツール関連](./troubleshooting.md#ai-tools) を参照してください。

## 基本的な使い方

### 対話の始め方

リポジトリルート（`/workspace`）で `claude` を起動し、日本語でそのまま依頼します。

```text
> 予約のステータス遷移がどこで実装されているか教えて
```

### CLAUDE.md 連携

Claude Code は起動時にリポジトリルートの [`CLAUDE.md`](https://github.com/CHS-Training-Org/ai_training_for_chuo_system/blob/main/CLAUDE.md) を自動で読み込みます。本リポジトリの `CLAUDE.md` には技術スタック・ディレクトリ構成・よく使うコマンド・コーディング規約が記載済みのため、**毎回これらを説明しなくても前提を踏まえた回答が得られます**。

!!! note "CLAUDE.md は「AI への申し送り事項」"
    プロジェクト固有の約束事（規約・コマンド・設計方針）を CLAUDE.md に書いておくと、すべてのセッションで一貫して反映されます。記載内容が古くなっていたら PR で更新してください。

### よく使う操作

| 操作 | 内容 |
|------|------|
| ++shift+tab++ | モード切替。**プランモード**にすると、コードを変更する前に計画を提示させて承認できる（大きめの変更で推奨） |
| ++esc++ | 実行中の応答を中断する |
| `/clear` | 会話履歴をリセットする（別のタスクに移るとき） |
| `/compact` | 長くなった会話を要約して続行する |
| `/help` | ヘルプを表示する |

## 効果的なプロンプトの書き方

3 つの原則を意識すると、回答の精度が大きく変わります。

1. **コンテキストを先に伝える** — 何をしようとしていて、どこまで分かっているか
2. **ゴールと制約を具体的に** — 完了条件・触ってよい範囲・守るべき規約
3. **仕様書を参照させる** — 本リポジトリでは `Docs/spec/` が真実の源。実装の根拠は仕様に求める

=== "Good な例"

    ```text
    予約一覧画面に「ステータスで絞り込むフィルタ」を追加したい。
    仕様は Docs/spec/screen-spec.md の予約一覧の節を参照して。
    バックエンド API（GET /api/reservations）は status クエリパラメータを
    すでにサポートしているので、フロントエンドのみ変更して。
    既存のリソース一覧のフィルタ実装と同じパターンに合わせること。
    ```

    → 目的・仕様の参照先・変更範囲・合わせるべき既存パターンが明確。

=== "Bad な例"

    ```text
    フィルタ機能を作って
    ```

    → どの画面か・何で絞り込むか・どこまで作るかが不明で、意図と違う実装になりやすい。

!!! tip "回答を鵜呑みにせず、根拠を確認する"
    「その仕様はどこに書いてある？」「変更したファイルの diff を見せて」のように根拠を確認する習慣をつけると、
    AI の誤り（ハルシネーション）に気づきやすくなります。

| # | 状況 | 使ったプロンプト（要約） | 結果・気づき |
|---|------|------------------------|--------------|
| 1 | コードベースの理解：予約申請〜承認の処理の流れを説明させる | （記入） | （記入） |
| 2 | 仕様の確認：`Docs/spec/requirements.md` を参照させてロールごとの権限を答えさせる | （記入） | （記入） |
| 3 | 影響範囲の調査：既存メソッドを 1 つ選び、変更した場合の影響箇所を調べさせる | （記入） | （記入） |
| 4 | テスト生成：既存の Service / Server Action を 1 つ選び、テストを生成させて実行する | （記入） | （記入） |
| 5 | エラー調査：意図的にコードを壊して、エラーメッセージから原因を特定させる | （記入） | （記入） |

## AI 利用ポリシー { #prohibited }

本リポジトリにおける AI ツール（Claude Code）の利用範囲と責任分担を定めます。STEP-03 に限らず、全課題に適用される恒久的なルールです。

### 利用範囲

| 用途 | 可否 |
|------|------|
| コード補完・実装支援 | ○ |
| 設計・仕様・エラーメッセージの解説 | ○ |
| テストコードの生成 | ○ |
| ドキュメント生成・要約 | ○ |
| 顧客情報・認証情報・社外秘文書の入力 | ✗ |
| 生成コードの無検証コミット | ✗ |

!!! danger "機密情報を入力しない"
    顧客情報・認証情報（パスワード・API キー・トークン）・社外秘の文書を、プロンプトやファイルとして AI に渡してはいけません。
    本リポジトリの学習範囲では実在の機密情報を扱いませんが、`.env.local` 等の値を会話に貼り付けない習慣をつけてください。

!!! danger "AI 生成コードを無検証でコミットしない"
    生成されたコードは必ず自分で読み、動作確認（テスト実行・画面確認）をしてからコミットしてください。
    理解できないコードをそのままコミットすることは禁止です。

### 責任とPRでの明記

!!! warning "生成コードの責任はマージした開発者が負う"
    「AI が書いたから」は不具合の言い訳になりません。PR を出した時点で、そのコードの品質はあなたの責任です。

PR には AI を活用した箇所の明記が**必須**です。[`.github/PULL_REQUEST_TEMPLATE.md`](../../.github/PULL_REQUEST_TEMPLATE.md) の「AI ツールを使った箇所」欄に、plan mode で提示された計画・メンター承認の有無・補完や生成に使った箇所を記入してください（詳しい手順は [dev-workflow.md](./dev-workflow.md#flow) を参照）。
