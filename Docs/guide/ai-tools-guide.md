# AI ツール活用ガイド（Claude Code）

このガイドは STEP-03「AI ツール導入・活用」の参照資料です。  
本リポジトリの標準 AI ツールである **Claude Code** の特性を理解し、状況に応じて意図的に活用できるようになることを目標とします。

---

## Claude Code とは

Claude Code は、ターミナル上で動作するコーディングエージェントです。  
エディタのインライン補完と異なり、**リポジトリ全体を読み取りながら対話形式でタスクを進められる**のが特徴です。

**得意な場面**

| 場面 | 例 |
|------|----|
| 広いコンテキストが必要な質問 | 「予約申請から承認までの処理がどのファイルを通るか説明して」 |
| 複数ファイル横断の変更 | 「`Resource` に `description` フィールドを追加して。Entity・DTO・画面まで一式」 |
| 影響範囲の安全確認 | 「このメソッドのシグネチャを変えたら、呼び出し元はどこが壊れる？」 |
| テスト・ドキュメント生成 | 「この Service のユニットテストを既存テストの書き方に合わせて作成して」 |
| エラー原因の調査 | エラーメッセージを貼り付けて「原因を調査して」 |

## セットアップ { #setup }

Claude Code は **DevContainer に組み込み済み**です。学習者が追加でインストールする必要はありません。

| 構成要素 | 内容 |
|---------|------|
| CLI | devcontainer feature（`anthropics/devcontainer-features/claude-code`）で自動インストール |
| VS Code 拡張 | `Anthropic.claude-code` が自動インストール |
| 認証情報 | `.claude`（`.credentials.json` を含む）はコンテナ専用の named volume に隔離。ホストとは共有しない |

### 初回の認証

DevContainer 内のターミナルで起動します。

```bash
claude --version   # インストール確認
claude             # 対話セッションを開始
```

初回起動時はログイン（ブラウザ認証）を求められます。画面の指示に従って認証してください。  
認証情報はコンテナ専用の named volume に保存されるため、**コンテナを再作成してもログインし直す必要はありません**（volume 自体を削除した場合を除く）。

!!! tip "ホストで Claude Code を使ったことがある場合でも、コンテナ内では初回ログインが必要"
    `.claude` ディレクトリはホストと共有せずコンテナ専用の named volume に隔離しているため、ホスト側の認証情報は引き継がれません。コンテナ内のターミナルで一度だけ `claude` にログインしてください。

!!! warning "コンテナ起動が `.claude.json` 関連で失敗する場合"
    マウント元の `~/.claude.json` はコンテナ初回起動時に自動生成されます（`devcontainer.json` の `initializeCommand`）。  
    起動に失敗する場合は [troubleshooting.md §AI ツール関連](./troubleshooting.md#ai-tools) を参照してください。

## 基本的な使い方

### 対話の始め方

リポジトリルート（`/workspace`）で `claude` を起動し、日本語でそのまま依頼します。

```text
> 予約のステータス遷移がどこで実装されているか教えて
```

### VS Code 拡張機能としての利用

ターミナルの `claude` コマンドに加えて、VS Code 拡張機能（`Anthropic.claude-code`、セットアップ時に自動インストール済み）からもグラフィカルパネルで同じ Claude Code を使えます。  
エディタ右上の Spark アイコン、またはコマンドパレットで「Claude Code」と入力して開きます。

拡張機能ならではの利点：

| 機能 | 内容 |
|------|------|
| 差分レビュー | Claude が提案する変更を、受け入れる前にエディタ上の差分表示で確認・編集できる |
| `@` メンション | ファイル名の一部やコード選択範囲を `@` で直接参照させられる |
| 複数会話 | 別のタブやウィンドウで複数の会話を並行して開ける |

ターミナルとパネルは会話履歴を共有しているため、`claude --resume` でパネル側の会話をターミナルから再開することもできます。

### CLAUDE.md 連携

Claude Code は起動時にリポジトリルートの [`CLAUDE.md`](https://github.com/CHS-Training-Org/ai_training_for_chuo_system/blob/main/CLAUDE.md) を自動で読み込みます。  
本リポジトリの `CLAUDE.md` には技術スタック、ディレクトリ構成、よく使うコマンド、コーディング規約が記載済みのため、**毎回これらを説明しなくても前提を踏まえた回答が得られます**。

!!! note "CLAUDE.md は「AI への申し送り事項」"
    プロジェクト固有の約束事（規約、コマンド、設計方針）を CLAUDE.md に書いておくと、すべてのセッションで一貫して反映されます。

### よく使う操作

| 操作 | 内容 |
|------|------|
| ++shift+tab++ | モード切替。**プランモード**にすると、コードを変更する前に計画を提示させて承認できる（大きめの変更で推奨） |
| ++esc++ | 実行中の応答を中断する |
| `/clear` | 会話履歴をリセットする（別のタスクに移るとき） |
| `/compact` | 長くなった会話を要約して続行する |
| `/help` | ヘルプを表示する |

## モデルの選択と使用量の管理 { #model-and-usage }

### モデルの種類と使い分け

Claude Code は複数のモデルから選んで実行できます。モデルごとに推論の精度と、応答速度や消費するトークン量のトレードオフが異なるため、タスクの性質に応じて選び分けます。

| モデル | 特徴 | 向いている場面 |
|-------|------|--------------|
| Opus | 複雑な推論に強いが、応答時間とトークン消費が大きい | 設計方針の検討、影響範囲が広い変更の計画 |
| Sonnet | 精度とコストのバランスが良く、標準的な選択肢 | 日常的な実装や修正のほとんど |
| Haiku | 最も高速で低コストだが、推論力は限定的で、単純なタスク向き | 定型的な小タスク、サブエージェントへの委譲 |

本リポジトリでは、**Sonnet を基本モデルとする方針**を採ります。日常的な実装や修正のほとんどは Sonnet の精度で足り、Opus に比べて応答時間とコストを抑えられるためです。  
`.claude/settings.json` の `model: sonnet` がこの方針を対話セッションの既定として反映しており、学習者が個別に切り替えない限り Sonnet が使われます（`advisorModel: opus` は `advisor` ツール専用の指定で、この既定とは別に扱われます。[agent-config.md §言語・モデル設定](../claude/agent-config.md#model-settings)参照）。

### モデルを切り替える

セッション中に `/model` コマンドを実行すると、対話形式でモデルを切り替えられます。  
基本モデル（Sonnet）から恒久的に変更したい場合は `/config` から設定します（設定は `settings.json` の `model` キーに保存されます）。

!!! tip "Opus に切り替える場面"
    STEP-03 のように既存パターンをなぞる実装では Sonnet で十分ですが、AI-DLC の INCEPTION フェーズ（[dev-workflow.md](./dev-workflow.md)）のように設計方針そのものを検討する場面では、Opus に切り替えると精度が上がることがあります。切り替えはそのタスクに限った一時的な操作とし、終わったら Sonnet に戻します。

`/fast` コマンドは Opus の応答を高速化します。トークン消費量自体が増えるわけではなく、応答時間を短縮する代わりに追加の利用枠を消費する仕組みなので、常用せず必要な場面に絞って使います。

### 使用量（利用上限）を確認する

Claude Code のサブスクリプション（Claude Pro プランと Max プラン）には、直近5時間の利用量に基づく制限と、7日間（週次）の利用量に基づく制限の2種類があります。  
どちらかの上限に達すると、その期間が経過してリセットされるまで新しい会話を送信できなくなります。

現在の消費量はセッション中に `/usage` コマンドで確認できます。  
直近のトークン消費量に加え、Pro プランと Max プランでは直近24時間分と直近7日間分の内訳を、  
スキルやサブエージェント、プラグイン、MCP サーバーごとの割合で表示します。

!!! note "本リポジトリの statusline での確認"
    `.claude/scripts/statusline-command.sh` は、5時間と7日のレートリミットを JST 表示で画面下部に常時表示する設定になっています（[agent-config.md §その他の設定](../claude/agent-config.md#other-config)）。  
    `/usage` を都度実行しなくても残量を把握できます。

## 効果的なプロンプトの書き方

3 つの原則を意識すると、回答の精度が大きく変わります。

1. **コンテキストを先に伝える**：何をしようとしていて、どこまで分かっているかを示す
2. **ゴールと制約を具体的に**：完了条件、触ってよい範囲、守るべき規約を示す
3. **仕様書を参照させる**：本リポジトリでは `Docs/spec/` が真実の源であり、実装の根拠は仕様に求める

=== "Good な例"

    ```text
    予約一覧画面に「ステータスで絞り込むフィルタ」を追加したい。
    仕様は Docs/spec/screen-spec.md の予約一覧の節を参照して。
    バックエンド API（GET /api/reservations）は status クエリパラメータを
    すでにサポートしているので、フロントエンドのみ変更して。
    既存のリソース一覧のフィルタ実装と同じパターンに合わせること。
    ```

    → 目的、仕様の参照先、変更範囲、合わせるべき既存パターンが明確に伝わる。

=== "Bad な例"

    ```text
    フィルタ機能を作って
    ```

    → どの画面か、何で絞り込むか、どこまで作るかが不明で、意図と違う実装になりやすい。

!!! tip "回答を鵜呑みにせず、根拠を確認する"
    「その仕様はどこに書いてある？」「変更したファイルの diff を見せて」のように根拠を確認する習慣をつけると、  
    AI の誤り（ハルシネーション）に気づきやすくなります。

さらに実践的なテクニック（検証の与え方、コンテキスト管理、よくある失敗パターンなど）は [claude-code-best-practices.md](./claude-code-best-practices.md) を参照してください。

## AI-DLC を使わずに最初の選択課題を進める（STEP-03） { #checklist }

STEP-03 では、[Beginner の課題](./enhancement-catalog.md#beginner) から選んだ最初の 1 課題を、**AI-DLC エンジン**（`/aidlc` が起動する INCEPTION フェーズ・承認ゲート・段階的な CONSTRUCTION フェーズの自動オーケストレーション）を使わずに進めます。  
狙いは、エンジンが計画・段階分割・承認ゲートを代行してくれる価値を、後続課題で AI-DLC を使ったときとの対比で体感することです。

AI-DLC エンジンを使わないことは、Claude Code を使わないことを意味しません。コード補完・設計相談・生成コードの相棒としては、この STEP でも通常どおり Claude Code を使います。  
同様に、**spec-first**（実装より先に `Docs/spec/` を更新する原則）とセルフレビューは、AI-DLC の有無にかかわらずリポジトリ全体の必須ルールなので、このステップでも維持します。

進め方は次のとおりです。

1. [coding-conventions.md §共通方針](./coding-conventions.md#common) の規約に従い、`feature/<GitHubユーザー名>/<issue番号>-<short-desc>` ブランチを作成する
2. 選んだ課題の要件シート（`Docs/spec/enhancements/<short-desc>.md`）を読み、変更が仕様に及ぶ範囲を `/update-spec` スキルで先に `Docs/spec/` に反映する
3. `/aidlc` は起動せず、Claude Code に実装したい内容を直接プロンプトする。実装順序や確認のタイミングは、エンジンの Workflow Planning に頼らず自分で組み立てる
4. 生成されたコードは必ず自分で読み、テスト実行・画面確認をしてからコミットする
5. PR を作成し `/draft-pr` で下書きを生成する。[review-criteria.md](./review-criteria.md) のチェックリストでセルフレビューし、満たしていることを確認したら自分でマージする

完了したら、AI-DLC なしで進めて手間だった点（段取り、仕様との整合、レビューの負荷など）を短く振り返り、その選択課題の PR に記載してください。後続の選択課題で `/aidlc` を使ったときに、この振り返りが対比の基準になります。

## AI 利用ポリシー { #prohibited }

本リポジトリにおける AI ツール（Claude Code）の利用範囲と責任分担を定めます。STEP-03 に限らず、全課題に適用される恒久的なルールです。

### 利用範囲

| 用途 | 可否 |
|------|------|
| コード補完・実装支援 | ○ |
| 設計・仕様・エラーメッセージの解説 | ○ |
| テストコードの生成 | ○ |
| ドキュメント生成・要約 | ○ |
| 顧客情報・認証情報・社外秘文書の入力 | ✗ |
| 生成コードの無検証コミット | ✗ |

!!! danger "機密情報を入力しない"
    顧客情報、認証情報（パスワード、API キー、トークン）、社外秘の文書を、プロンプトやファイルとして AI に渡すわけにはいきません。  
    本リポジトリの学習範囲では実在の機密情報を扱いませんが、`.env.local` 等の値を会話に貼り付けない習慣をつけてください。

!!! danger "AI 生成コードを無検証でコミットしない"
    生成されたコードは必ず自分で読み、動作確認（テスト実行、画面確認）をしてからコミットしてください。  
    理解できないコードをそのままコミットするわけにはいきません。

### 責任とPRでの明記

!!! warning "生成コードの責任はマージした開発者が負う"
    「AI が書いたから」は不具合の言い訳になりません。PR を出した時点で、そのコードの品質はあなたの責任です。

PR には AI を活用した箇所の明記が**必須**です。  
[`.github/PULL_REQUEST_TEMPLATE.md`](../../.github/PULL_REQUEST_TEMPLATE.md) の「AI ツールを使った箇所」欄に、plan mode で提示された計画、補完や生成に使った箇所を記入してください（詳しい手順は [dev-workflow.md](./dev-workflow.md#flow) を参照）。
