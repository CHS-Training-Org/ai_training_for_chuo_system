# Claude Code ベストプラクティス

このガイドは、Claude Code 公式サイトの「[ベストプラクティス](https://code.claude.com/docs/ja/best-practices)」から、学習者にとって特に有用な部分を抜粋し整理したものです。  
セットアップや基本操作、プロンプトの3原則は [ai-tools-guide.md](./ai-tools-guide.md) で扱っているため、このガイドはそれを読んだうえで一段深く使いこなすための補足として位置づけます。

原文にはセッションの並列運用やチームでの大規模運用を前提にした内容も含まれますが、そうした範囲はここでは取り上げません。

---

## コンテキストウィンドウを意識する

Claude Code とのやり取りは、すべて**コンテキストウィンドウ**という有限の領域に積み重なります。会話のメッセージ、読み取ったファイルの内容、実行したコマンドの出力は、すべてこの領域を消費します。

コンテキストウィンドウが埋まってくると、Claude の回答精度は下がります。  
以前の指示を忘れたり、無関係な情報に引きずられて誤った判断をしたりしやすくなるためです。したがって、コンテキストウィンドウは Claude Code を使う際に管理すべき最も重要なリソースだと捉えてください。

!!! tip "会話をリセットする"
    関係のないタスクに移るときは `/clear` で会話履歴をリセットします。長くなった会話を要約して続けたい場合は `/compact` を使います。

## Claude に自分の作業を検証する方法を与える

Claude は、作業が完了したように見えた時点で手を止めます。  
実行して確認できるチェック（テスト、ビルド、比較用のスクリーンショットなど）を与えられていないと、「完了したように見える」ことが唯一の判断材料になり、検証の役割をすべて自分で担うことになります。

Claude が実行してパス・フェイルを判定できるものを与えると、Claude 自身がチェックを実行し、結果を読み、パスするまで修正を繰り返すという流れが成立します。

=== "検証基準を与えない例"

    ```text
    メールアドレスを検証する関数を実装する
    ```

=== "検証基準を与える例"

    ```text
    validateEmail 関数を書く。テストケースの例：
    user@example.com は true、invalid は false、user@.com は false。
    実装後にテストを実行する。
    ```

!!! tip "成功を主張させず、証拠を示させる"
    「できました」という報告だけでなく、テストの実行結果や実行したコマンドとその戻り値を示すよう求めてください。  
    証拠を確認する方が、自分で検証をやり直すより速く、しかも見ていなかった作業でも機能します。

## 探索 → 計画 → 実装 → コミットの順で進める

Claude にいきなり実装させると、間違った問題を解決してしまうことがあります。調査・計画を実装から分離すると、この失敗を避けられます。おすすめの流れは次の4段階です。

1. **探索**：プランモードに入り、Claude にファイルを読み取らせて質問に答えさせる（変更は加えさせない）
2. **計画**：詳細な実装計画を作成させる
3. **実装**：プランモードを終了し、計画に沿って実装させ、テストで検証させる
4. **コミット**：説明的なメッセージでコミットし、PR を作成させる

!!! note "計画をスキップしてよいとき"
    プランモードには相応のオーバーヘッドがあります。スコープが明確で修正が小さいタスク（タイプミスの修正、ログ出力の追加など）では、計画を挟まずに直接実行させて構いません。  
    計画が役立つのは、進め方に確信が持てないとき、変更が複数ファイルに及ぶとき、変更対象のコードに不慣れなときです。差分を一文で言い切れるなら計画は不要です。

## プロンプトに具体的な文脈を与える

指示が具体的であるほど、後から直す量が減ります。ファイルを名前で指定し、制約を明示し、参考にすべき既存のパターンを示してください。

| 戦略 | 悪い例 | 良い例 |
|------|--------|--------|
| タスクの範囲を絞る | 「foo.py のテストを追加する」 | 「foo.py に、ユーザーがログアウトしているエッジケースを検証するテストを書く。モックは使わない」 |
| 既存のパターンを参照させる | 「カレンダーウィジェットを追加する」 | 「ホームページの既存ウィジェット（`HotDogWidget.php` が好例）の実装を確認し、同じパターンで月選択・年ページネーションができるウィジェットを実装する」 |
| 症状を説明する | 「ログインバグを修正する」 | 「セッションタイムアウト後にログインが失敗するとの報告がある。`src/auth/` の認証フロー、特にトークン更新を確認する。再現する失敗テストを書き、修正する」 |

ファイルの場所を口頭で説明する代わりに `@ファイル名` で直接参照させると、Claude は応答前にその内容を読み込みます。  
エラーログを渡すときは `cat error.log | claude` のようにパイプで直接渡す方法もあります。

## 効果的な CLAUDE.md を書く

`CLAUDE.md` に何を書くべきかは、次の基準で判断します。

| 含めるべき内容 | 除外すべき内容 |
|---|---|
| Claude が推測できない Bash コマンド | コードを読めば分かること |
| デフォルトと異なるコードスタイルの規約 | 既に知っている標準的な言語規約 |
| テストの実行方法や推奨するテストランナー | 詳細な API ドキュメント（代わりにリンクする） |
| ブランチ命名や PR 規約などのリポジトリの慣習 | 頻繁に変わる情報 |
| 必須の環境変数など、開発環境固有の癖 | 長い説明やチュートリアル |

判断に迷う行があれば、「この行を削除したら Claude は間違えるか」と自問してください。答えが「いいえ」なら、その行は削除して構いません。  
`CLAUDE.md` が肥大化すると、重要なルールがノイズに埋もれて逆に無視されやすくなります。

`CLAUDE.md` はホームフォルダ（`~/.claude/CLAUDE.md`、全セッション共通）、プロジェクトルート（`./CLAUDE.md`、チーム共有）、`./CLAUDE.local.md`（個人用、`.gitignore` 対象）、モノレポの親子ディレクトリなど、複数の場所に置けます。また `@path/to/import` という記法で、別ファイルの内容を読み込むこともできます。

## 早期かつ頻繁に方向転換する

会話は後から取り消せます。Claude が見当違いの方向に進んでいると気づいたら、すぐに修正してください。  
同じ問題を2回以上直している場合、コンテキストは失敗したアプローチで乱れています。そこで無理に続けるより、`/clear` して学んだことを踏まえた具体的なプロンプトで新しく始める方が、多くの場合早く正しい結果に着地します。

| 操作 | 効果 |
|------|------|
| ++esc++ | 実行中の応答を中断する（コンテキストは保持されるので、そのまま指示し直せる） |
| ++esc+esc++ または `/rewind` | 巻き戻しメニューを開き、以前の会話やコードの状態に戻す |
| 「元に戻して」と伝える | Claude 自身に変更を取り消させる |

!!! warning "巻き戻しは Git の代わりにはならない"
    巻き戻し機能が追跡するのは Claude が行った変更だけであり、外部のプロセスは対象外です。こまめな commit は別途必要です。

## よくある失敗パターンを避ける

| パターン | 症状 | 対処 |
|---|---|---|
| キッチンシンクセッション | 1つのタスクの途中で関係のない質問を挟み、また元のタスクに戻る | 関係のないタスクに移る前に `/clear` する |
| 何度も直す | Claude の間違いを直しても、また間違える | 2回直しても改善しなければ `/clear` し、学んだことを踏まえて最初から具体的に指示する |
| 過度に指定された `CLAUDE.md` | ルールが長すぎて、重要な指示が埋もれて無視される | 容赦なく削除する。指示なしで既に正しくできていることは書かない |
| 信頼してから検証するギャップ | もっともらしく見えるが、エッジケースを処理していない実装が生成される | 検証手段（テスト・スクリプト・スクリーンショット）を必ず与える。検証できないものは出荷しない |
| 無限探索 | スコープを決めずに「調査して」と依頼し、Claude が大量のファイルを読んでコンテキストを消費する | 調査の範囲を狭く指定する |

## 直感を育てる

ここで紹介したパターンは、あらゆる状況に最適というわけではありません。  
複雑な1つの問題に深く入り込んでいる間は、コンテキストを蓄積させたままの方がよいこともあります。探索的なタスクでは、あえて曖昧なプロンプトのままにしておき、Claude がどう解釈するかを先に見た方がよいこともあります。

Claude の出力がうまくいったときは、何が効いたのか（プロンプトの具体性、与えた文脈、使っていたモード）に注意を払ってください。逆にうまくいかなかったときは、その原因（コンテキストのノイズ、プロンプトの曖昧さ、タスクの大きさ）を考えてみてください。この振り返りの積み重ねが、ガイドだけでは得られない実践的な判断力になります。
